# vision: declastruct-squarespace domain transfer

## the outcome world

### day-in-the-life

**before:**
you have 300+ domains trapped in squarespace. no api. to transfer each domain out manually:
1. log in to squarespace (2fa every time)
2. navigate to domain settings
3. disable dnssec
4. unlock the domain
5. request transfer code
6. wait for email
7. repeat 300+ times

that's ~6-10 minutes per domain. 50+ hours of clicking. pure pain.

**after:**
```ts
// resources.ts
export const getResources = async () => {
  const provider = getDeclastructSquarespaceProvider({
    account: { id: 'my-account', email: 'me@company.com' },
    credentials: {
      email: process.env.SQUARESPACE_EMAIL,
      password: process.env.SQUARESPACE_PASSWORD,
      totpSecret: process.env.SQUARESPACE_TOTP_SECRET,
    },
  });

  // get all domains (context baked into provider - no need to pass explicitly)
  const domains = await provider.daos.DeclaredSquarespaceDomainRegistration.get.all({});

  // declare desired state: unlock all, request transfer codes
  return domains.flatMap((domain) => [
    new DeclaredSquarespaceDomainRegistration({
      ...domain,
      isLocked: false,
      dnssecEnabled: false,
    }),
    new DeclaredSquarespaceDomainTransferRequest({
      domain: refByUnique(domain),  // typed ref, not bag ref
    }),
  ]);
};
```

then:
```sh
npx declastruct plan --wish resources.ts --into plan.json
npx declastruct apply --plan plan.json
```

go get coffee. all 300 domains unlocked, transfer codes requested. emails arrive. done.

### the "aha" moment

the moment you realize: *"i can treat squarespace domains like terraform resources"*

- declare what you want
- compare to reality via plan
- apply changes idempotently
- re-run safely anytime

no more fear of clicking the wrong button on domain 247.

## user experience

### usecases

| goal | contract |
|------|----------|
| see all my domains | `daos.DeclaredSquarespaceDomainRegistration.get.all()` |
| check dns records | `daos.DeclaredSquarespaceDomainDnsRecord.get.all({ domain: { name } })` |
| unlock a domain | `daos.DeclaredSquarespaceDomainRegistration.set.upsert({ entity: { name, isLocked: false } })` |
| disable dnssec | `daos.DeclaredSquarespaceDomainRegistration.set.upsert({ entity: { name, dnssecEnabled: false } })` |
| request transfer code | `daos.DeclaredSquarespaceDomainTransferRequest.set.findsert({ entity: { domain: refByUnique(domain) } })` |
| batch transfer 300 domains | declare all via resources file, `declastruct apply` |

### typical workflow timeline

| phase | time | what happens |
|-------|------|--------------|
| setup | 5 min | install package, configure credentials |
| plan | ~30 sec | compare declared state vs squarespace reality |
| review | 2 min | inspect plan.json, verify changes look right |
| apply | 1-3 hours | playwright automates all UI interactions (batch sequential) |
| verify | 1 min | re-plan to confirm all changes applied (all KEEP) |

total: ~1-3 hours automated for 300 domains vs 50+ hours manually

### contract inputs/outputs

**provider creation:**
```ts
const provider = getDeclastructSquarespaceProvider({
  account: { id: string, email: string },
  credentials: { email, password, totpSecret },  // totpSecret REQUIRED - sms/passkey 2fa not supported
  cache?: { directory?: string },
  browser?: { wsEndpointExtant?: string },
});
```

**what you get:**
```ts
// daos with context baked in - no need to pass context on each call
provider.daos.DeclaredSquarespaceDomainRegistration  // domain operations
provider.daos.DeclaredSquarespaceDomainDnsRecord     // dns record operations
provider.daos.DeclaredSquarespaceDomainTransferRequest  // transfer operations
```

## evaluation

### how well does it solve the goals?

| goal | solution |
|------|----------|
| enumerate domains | `get.all()` scrapes full list via playwright |
| see dns records | `get.all({ domain })` scrapes dns settings page |
| unlock domains | `set.upsert({ isLocked: false })` toggles lock via UI |
| request transfer codes | `set.findsert()` clicks through transfer flow |
| batch operations | declare all, apply once |
| idempotent | findsert/upsert semantics, re-runnable safely |

### pros

- **no api required** — playwright automates the UI squarespace provides
- **stealth mode** — TODO: add `playwright-extra` + `puppeteer-extra-plugin-stealth` to reduce bot detection risk (currently only userAgent is set)
- **declarative** — state what you want, not how to get there
- **idempotent** — safe to re-run, compares desired vs actual
- **batch-friendly** — 1 or 300 domains, same workflow
- **cached** — remote state cached to avoid re-scraping
- **familiar** — follows terraform mental model

### cons

- **ui fragility** — squarespace ui changes could break selectors
- **rate limits** — too many operations too fast might trigger squarespace defenses
- **browser overhead** — playwright requires chromium, headless mode
- **no true api** — we're automating a UI, not calling an api
- **email-delivered codes** — transfer codes go to email, not returned directly

### pit-of-success edgecases

| edgecase | handling |
|----------|----------|
| domain is locked for 60-day period | `lockReason` field indicates why, prevents futile unlock attempts |
| domain already unlocked | upsert returns current state, no-op |
| transfer already requested | findsert returns extant request, no duplicate email |
| stale cache | cache TTL, can be cleared via config |
| 2fa required | totpSecret generates codes automatically |
| bot detection | TODO: add stealth plugin; currently only userAgent + 500ms delay |

## open questions & assumptions

### assumptions made

1. **squarespace ui selectors remain stable** — we've encoded specific selectors; ui redesigns would require updates
2. **one account at a time** — current design assumes single account per provider instance
3. **transfer codes via email is acceptable** — no programmatic code retrieval
4. **chromium-based automation is acceptable** — no lightweight http-only option extant
5. **stealth plugin required but absent** — must add `playwright-extra` + `puppeteer-extra-plugin-stealth` before production use; currently only userAgent is set
6. **wish typo noted** — wish line 3 says "cloudflare" but means "squarespace" (confirmed by rest of wish)

### questions for the wisher

1. ~~**multiple accounts?**~~ — one account per run is fine (confirmed)
2. ~~**target registrar integration?**~~ — out of scope, handled in declastruct-cloudflare (confirmed)
3. ~~**monitor/alert?**~~ — out of scope, squarespace sends emails for transfer codes and status (confirmed)
4. ~~**parallelism?**~~ — sequential preferred for safety (confirmed)

### must validate

- [ ] playwright selectors work against current squarespace ui
- [ ] 2fa totp integration works reliably
- [ ] cache TTL is appropriate for 300+ domain accounts
- [ ] no squarespace rate limits triggered during batch operations
- [ ] different registrar types (SQUARESPACE_DOMAINS_LLC, TUCOWS, KEY_SYSTEMS) have same transfer UI flow
- [x] 2fa is always totp (not sms or security key) — **CONSTRAINT**: squarespace supports totp, sms, and passkeys; only totp is automatable; users MUST configure their account to use authenticator app (totp) for automation to work; sms and passkey 2fa are NOT SUPPORTED
- [ ] sessions don't timeout during multi-hour batch (300 domains)
- [ ] all domains are transferable (none in grace period, disputed, or locked)
- [ ] no captcha challenges during automation

### anti-patterns to avoid (the inverse)

| outcome we must avoid | how we prevent it |
|----------------------|-------------------|
| domains stuck in limbo (unlocked but transfer incomplete) | transfer request findsert ensures code is requested after unlock |
| accidental mass-unlock of critical domains | plan review before apply; user declares which domains explicitly |
| account lockout from automation detection | sequential processing with delays; bottleneck rate limiting |
| transfer codes expire before use | document expiration; user must act promptly on emails |
| dns outages during transfer window | out of scope (dns propagation is registrar-to-registrar concern) |
| partial batch failure with inconsistent state | idempotent operations allow safe re-run; no rollback needed |
| no way to filter/exclude specific domains | user controls resources.ts declarations; can filter in code |
| multiple code requests invalidate prior codes | findsert semantics prevent duplicate requests |
| 15-day transfer timeline causes confusion | document timeline; transfer completion is external to this tool |
| 5-day grace period rejection window missed | document that rejections must happen within 5 days via email |
| credentials leaked in error logs | redact password and totpSecret from all log output |
| browser resource exhaustion in batch | reuse browser context; clean up pages after each domain |
| silent failure on selector mismatch | fail fast with clear error; do not return empty data silently |
| batch halts mid-run and can't resume | checkpoint progress; enable resumable batches |
| rate limit detection via patterns | randomized delays between operations; vary intervals |
| email deliverability issues at scale | batch in smaller groups; verify email receipt before next batch |
| locked domains hidden in plan output | plan must prominently highlight untransferable domains |
| session expires mid-batch | health check before each operation; auto-re-login on expiry |
| critical domains frozen by TOS ban | transfer critical/high-value domains first; have escalation path |

## what is awkward?

### feels off

1. **no true api** — we're screen-scraping, which feels fragile compared to a real api. but squarespace gives us no choice.

2. **transfer code in email** — the transfer authorization code gets emailed, not returned. user must check email separately. this breaks the fully-automated dream slightly.

3. **domain registration vs transfer request split** — two entities for one conceptual "transfer this domain" action. necessary because they're separate ui flows, but adds cognitive overhead.

### fights mental model

- users might expect `transfer(domain)` as single action, but we separate "prepare domain for transfer" (unlock, disable dnssec) from "request transfer code"
- justified: these are genuinely separate steps with different failure modes

### uncomfortable tradeoffs

| tradeoff | chosen | alternative |
|----------|--------|-------------|
| ui automation vs api | ui (no api extant) | none |
| sequential vs parallel | sequential (safe) | parallel (faster, riskier) |
| cache vs fresh | cached (performant) | always-fresh (slower, accurate) |
| playwright vs puppeteer | playwright (modern, better api) | puppeteer (more mature) |

### devil's advocate — why this might be wrong

| criticism | response |
|-----------|----------|
| "just do it manually — it's only 50 hours" | 50 hours of tedious clicks vs 3 hours of automated batch; engineer time is expensive |
| "wait for squarespace to release an api" | no indication they will; they've had years; we have 300 domains to move now |
| "ui automation is inherently brittle" | true — but selectors use data-test attributes with fallbacks; maintenance burden is real but acceptable |
| "this solves a one-time problem" | yes, but the pattern (declastruct + playwright) is reusable for other ui-only platforms |
| "squarespace might detect and block automation" | sequential with delays mimics human behavior; no evidence of bot detection yet |

## summary

declastruct-squarespace transforms a 50+ hour manual nightmare into a 1-3 hour automated workflow. users declare desired state, compare via plan, and apply changes idempotently. it's terraform for squarespace domains — built on playwright because squarespace refuses to provide a real api.
