### dao contract

```ts
interface DeclastructDao<T> {
  get: {
    one: {
      byUnique(input: { unique: RefByUnique<T> }, context): Promise<T | null>;
    };
    all(input: {}, context): Promise<T[]>;
  };
  set: {
    findsert?(input: { entity: T }, context): Promise<T>;
    upsert?(input: { entity: T }, context): Promise<T>;
    delete?(input: { unique: RefByUnique<T> }, context): Promise<void>;
  };
}
```

## design decisions (applied)

the following documents design decisions already implemented in the codebase.

### decision 1: DeclaredSquarespaceDomainDnsRecord primary key semantics

**file:** `src/domain.objects/DeclaredSquarespaceDomainDnsRecord.ts`

**implemented:**
```ts
public static primary = ['domain'] as const;
public static unique = ['domain', 'type', 'host'] as const;
```

**rationale:** for child entities (DNS records belong to domains):
- `primary = ['domain']` = parent reference key (which domain does this record belong to?)
- `unique = ['domain', 'type', 'host']` = identity key (which specific record is this?)

this pattern enables `RefByPrimary` to reference the parent domain while `RefByUnique` references a specific record.

**test verification:**
```ts
then('primary key is parent domain reference', () => {
  expect(DeclaredSquarespaceDomainDnsRecord.primary).toEqual(['domain']);
});
```

---

## decision 10: session health check (applied)

long batches (3+ hours for 300 domains) risk session expiry mid-operation.

**implemented in:**
- `src/access/sdks/squarespace.via.playwright/auth/checkSessionHealth.ts` - checks session validity
- `src/access/sdks/squarespace.via.playwright/wrappers/withNewLoggedInBrowserPage.ts` - calls health check before each operation

**flow:**
```ts
// before each operation
const sessionStatus = await checkSessionHealth(page);
if (!sessionStatus.valid) {
  // session expired - re-login on current page
  await performSquarespaceLogin(page, agentOptions.credentials);
}
```

**credential failure recovery:**
- password changed → update env var, resume batch
- TOTP secret reset → update env var, resume batch
- account locked → wait for unlock, resume batch
- idempotent operations mean "resume" = "re-run command"; completed domains skipped

**rationale:** prevents batch failures from session timeout. auto re-login keeps batch active.

## anti-patterns to avoid

these anti-patterns were identified in inverse review and documented to prevent regression.

### must not: parallel operations

concurrent playwright operations would trigger squarespace bot detection and rate limits.

**current protection:** bottleneck with `maxConcurrent: 1` enforces sequential execution.

### must not: barrel exports

`export * from './file'` patterns create circular import risks.

**current protection:** no barrel exports in daos or domain.operations (per rule.forbid.barrel-exports).

### must not: untyped bag refs

`{ name: domain.name }` loses type safety and intellisense.

**current protection:** all domain refs use `RefByUnique<typeof Entity>` or `RefByUnique.as<>()` (per rule.forbid.dobj.bagrefs).

### must not: mocks in tests

`jest.mock()` creates false confidence and drift from reality.

**current protection:** integration tests use real squarespace; unit tests use dependency injection.

### must not: error swallow

try/catch blocks that hide errors mask real failures.

**current protection:** error handlers rethrow after cleanup or return explicit failure status.

## known limitations

these limitations were surfaced in self-review and documented for transparency.

### fragile timeout in requestTransferCode

`requestTransferCode.ts` uses a fixed 3000ms timeout instead of explicit element waits:

```ts
// .note - fixed timeout because success elements vary by registrar
// .todo - replace with waitForSelector for known success indicators
await page.waitForTimeout(3000);
```

**risk:** timeout may be too short/long for different registrar types or network conditions.
**mitigation:** integration tests would catch failures. `.todo` marks this for future fix.

### ui selector drift risk

selectors use `data-test` attributes with fallbacks to text-based selectors:

```ts
requestCodeButton: '[data-test="request-code"], button:has-text("Transfer")'
```

**risk:** squarespace ui changes could break selectors.
**mitigation:** fallback patterns + integration tests catch drift.
**recovery:** when selectors break mid-batch: run integration tests to identify failure, update selector files (centralized in `selectors/*.ts`), test single domain, resume batch (idempotent).

### bot detection resilience (complete)

stealth plugin and captcha handle are implemented per factory blueprint.

**implemented:**
- `stealthChromium.ts` — playwright-extra with puppeteer-extra-plugin-stealth
- `detectCaptchaChallenge.ts` — turnstile iframe detection
- `clickTurnstileCheckbox.ts` — headful checkbox click
- `waitForHumanSignal.ts` — human fallback for puzzle
- `answerCaptchaChallenge.ts` — tiered orchestration
- `BrowserAuthSession.respawn()` — headless ↔ headful mode switch

**tiered approach:**
1. tier 1: stealth plugin reduces captcha trigger rate
2. tier 2: headful pivot + checkbox click (automatic)
3. tier 3: human fallback for puzzle challenges (rare)

**verification:** `ls src/access/sdks/squarespace.via.playwright/captcha/` returns 8 files.

### TOS considerations

automation against squarespace UI may conflict with Terms of Service. mitigations: sequential with delays respects infrastructure; manages own domains only; no competitive data harvest.

**per `.behavior/v2025_12_17.domain-transfer/3.3.0.blueprint.factory.v1.i1.md`:**

session persistence via playwright storageState is now implemented:
- `ContextSquarespaceAgentOptions.session.storageStatePath` config option added
- `genLoggedInBrowser.ts` restores storageState from disk if file extant
- `genLoggedInBrowser.ts` checks session health before login
- `genLoggedInBrowser.ts` saves storageState after successful login

**usage:**
```ts
const provider = getDeclastructSquarespaceProvider({
  account: { id: 'my-account', email: 'me@company.com' },
  credentials: { email, password, totpSecret },
  session: {
    storageStatePath: '.cache/squarespace-session.json',
  },
});
```

**benefits:**
- session reused across test runs (no login if session valid)
- minimizes login frequency → reduces bot detection risk
- minimizes rate limit risk for batch operations
