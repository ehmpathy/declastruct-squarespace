# research: factory blockers

## test infra

**current state**: integration tests exist for scrapers
- `scrapeDomainsList.integration.test.ts`
- `scrapeDomainDetail.integration.test.ts`
- `scrapeDnsRecords.integration.test.ts`
- `scrapeTransferRequests.integration.test.ts`

**sufficient**: yes, test infra covers all scraper operations

**no provision required**: tests run against live squarespace account

---

## feedback loops

**current bottleneck**: playwright browser automation is inherently slow (~2-5 sec per operation)

**mitigation**: remote state cache reduces repeated scrapes

**no automation gap**: all verification is automated via integration tests

---

## access and credentials

**required credentials**:
- `SQUARESPACE_EMAIL` — account email
- `SQUARESPACE_PASSWORD` — account password
- `SQUARESPACE_TOTP_SECRET` — for automated 2fa

**current state**: credentials documented in vision, env vars expected

**constraint**: user must configure totp (not sms or passkey)

**who provisions**: end user (their squarespace account)

---

## infra control

**no new infra required**

playwright runs locally (headless chromium)

remote state cache uses filesystem (`simple-on-disk-cache`)

---

## other blockers

### squarespace ui stability

**risk**: selectors may break if squarespace redesigns ui

**mitigation**: selectors use data-test attributes with fallbacks

**impact**: maintenance burden, not a blocker

### rate limits

**risk**: batch operations may trigger rate limits

**mitigation**: sequential process with delays

**current status**: no evidence of rate limits yet

### test domains

**need**: domains to test transfer workflow end-to-end

**constraint**: domain transfers are irreversible

**mitigation**: test unlock/lock idempotency, skip actual transfer initiation in tests
