# test framework infrastructure research

## overview

this research documents the infrastructure patterns and techniques that will enable rapid feedback loops for testing the Squarespace domain transfer behavior. given the unique challenges of Playwright-based UI automation against a third-party service with no API, special attention is paid to strategies that minimize test execution time while maximizing confidence.

---

## test types and their purposes

### 1. unit tests (`*.test.ts`)

**purpose:** fast validation of domain logic in isolation

**characteristics:**
- no real browser interactions
- no network calls
- fully mocked dependencies
- runs in milliseconds per test
- executes via `npm run test:unit`

**example uses for this project:**
- domain object validation (DeclaredSquarespaceDomainRegistration, DeclaredSquarespaceDomainDnsRecord)
- casting/transformation logic (castToDeclaredSquarespaceDomainRegistration)
- selector parsing logic
- cache key generation

**feedback loop time:** < 1 second per test file

---

### 2. integration tests (`*.integration.test.ts`)

**purpose:** validate boundary components against real remote dependencies

**characteristics:**
- real Playwright browser sessions
- real Squarespace authentication
- real UI interactions
- tests individual operations in isolation
- executes via `npm run test:integration`

**boundary components to test:**
- `scrapeDomainsList` - scrapes domains list page
- `scrapeDomainDetail` - scrapes single domain details
- `scrapeDnsRecords` - scrapes DNS records for a domain
- `toggleDomainLock` - unlocks domain for transfer
- `requestTransferCode` - initiates transfer code request

**feedback loop time:** 10-60 seconds per test (browser launch + login + page interactions)

---

### 3. acceptance tests (`*.acceptance.test.ts`)

**purpose:** prove end-to-end behavior matches wish, vision, and criteria

**characteristics:**
- exercises full declastruct CLI workflow
- plan → apply → verify idempotency
- black-box testing (only contract layer visible)
- validates 300+ domain scale requirements
- executes via `npm run test:acceptance`

**feedback loop time:** 5-15 minutes for full acceptance suite

## playwright-specific testing challenges

### 1. authentication overhead

**problem:** every test run requires Squarespace login, potentially with TOTP 2FA

**mitigation strategies:**
- reuse browser context across tests via `useBeforeAll`
- persist auth state to disk between runs (cookies, localStorage)
- consider headless persistent browser daemon for development

```ts
// example: shared browser context
const scene = useBeforeAll(async () => {
  const browser = await findOrCreateLoggedInBrowser({ credentials });
  const page = await browser.newPage();
  return { browser, page };
});
```

### 2. flaky UI elements

**problem:** Squarespace UI elements may change, have dynamic IDs, or load asynchronously

**mitigation strategies:**
- use semantic selectors where possible (`[data-test-id]`, `role=`, `text=`)
- centralize selectors in dedicated modules (already done: `selectors/*.ts`)
- add explicit waits for element stability
- screenshot on failure for debugging

### 3. rate limiting

**problem:** rapid automated requests may trigger rate limiting or captchas

**mitigation strategies:**
- `Bottleneck` rate limiter already integrated in agentOptions
- add delays between actions
- use cached results for repeated reads

### 4. slow execution

**problem:** browser automation is inherently slow (seconds per page)

**mitigation strategies:**
- remote state caching (already implemented via `withRemoteStateCache`)
- parallel test execution where independent (jest `maxWorkers`)
- `--changedSince=main` for incremental test runs
- mocked unit tests for logic validation

## rapid feedback loop strategies

### strategy 1: layered testing pyramid

```
        /\
       /  \  acceptance (slow, end-to-end)
      /----\
     /      \ integration (medium, boundary)
    /--------\
   /          \ unit (fast, isolated)
  /------------\
```

**execution order for rapid feedback:**
1. unit tests first (catch logic errors in < 1 second)
2. integration tests for changed components
3. acceptance tests before merge/release

### strategy 2: selective test execution

**via jest `--changedSince` flag (already configured):**
```bash
# only run tests affected by changes since main
npm run test:unit        # uses --changedSince=main by default
npm run test:integration # uses --changedSince=main by default

# thorough run: all tests regardless of changes
THOROUGH=true npm run test:unit
THOROUGH=true npm run test:integration
```

### strategy 3: remote state caching

**already implemented in `src/infra/performance/withRemoteStateCache.ts`:**
- caches query results (domain list, DNS records)
- invalidates on mutations
- uses on-disk cache for persistence across runs

**benefits for testing:**
- second run of `getAllDomains` uses cache (ms vs 10s+)
- speeds up both integration and acceptance tests
- enables offline development for cached domains

### strategy 4: mocked vs real context factories

**mocked context (`getMockedSquarespaceContext`):**
- for unit tests
- no browser, no network
- instant execution

**real context (`getSampleSquarespaceContext`):**
- for integration/acceptance tests
- requires environment variables
- real browser, real authentication

### strategy 5: test data isolation

**approach:**
- use dedicated test domains (not production critical)
- test against domains already unlocked (for read-only tests)
- test against domains with auto-renew off (safe to manipulate)

## agent interface comparison

### robot:claude via claude-code

**interface:**
- CLI commands via Bash tool
- environment variables for credentials
- reads test output from stdout/stderr
- uses `| tee` pattern for log persistence (per `.skills/test.integration.sh`)

**workflow:**
```bash
# set credentials
export SQUARESPACE_EMAIL="..."
export SQUARESPACE_PASSWORD="..."
export SQUARESPACE_TOTP_SECRET="..."

# run specific test
npm run test:integration -- scrapeDomainsList.integration.test.ts

# run with thorough flag
THOROUGH=true npm run test:integration
```

**advantages:**
- can run tests in background
- can parse test output for failures
- can retry on transient failures

**limitations:**
- no visual debugging (headless only)
- limited to CLI interaction
- must interpret error messages from text output

### human:author via vscode/terminal

**interface:**
- same CLI commands
- IDE integration (test explorer, debugger)
- visual inspection of browser during headed runs

**workflow:**
```bash
# same commands as robot:claude
npm run test:integration -- scrapeDomainsList.integration.test.ts

# headed mode for debugging
PWDEBUG=1 npm run test:integration -- scrapeDomainsList.integration.test.ts
```

**advantages:**
- interactive debugging with breakpoints
- visual verification of browser state
- IDE test runner integration
- `PWDEBUG=1` for Playwright inspector

**limitations:**
- requires manual attention
- slower than automated runs

### interface parity

both agents use:
- same npm scripts
- same environment variables
- same jest configurations
- same test file patterns

**key difference:** human can use headed browser + debugger; robot is headless-only

## test framework configuration

### jest.unit.config.ts
- matches `**/*.test.ts` excluding integration/acceptance
- uses mocked dependencies
- 50% workers for parallelism
- < 1 second per test

### jest.integration.config.ts
- matches `**/*.integration.test.ts`
- 90 second timeout for browser operations
- requires AWS/Squarespace credentials
- runs against real browser/Squarespace

### jest.acceptance.config.ts
- matches `**/*.acceptance.test.ts`
- 90 second timeout
- runs in band (`--runInBand`) for sequential execution
- tests full declastruct CLI workflow

## recommended test patterns

### pattern 1: given-when-then with useBeforeAll

```ts
import { given, when, then, useBeforeAll } from 'test-fns';

describe('scrapeDomainsList', () => {
  given('a logged-in browser session', () => {
    const scene = useBeforeAll(async () => {
      const context = await getSampleSquarespaceContext();
      return { context };
    });

    when('scraping the domains list page', () => {
      then('returns all domains in the account', async () => {
        const domains = await scrapeDomainsList({}, scene.context);
        expect(domains.length).toBeGreaterThan(0);
      });

      then('each domain has required fields', async () => {
        const domains = await scrapeDomainsList({}, scene.context);
        domains.forEach(d => {
          expect(d.name).toBeDefined();
          expect(d.status).toBeDefined();
        });
      });
    });
  });
});
```

### pattern 2: data-driven unit tests

```ts
const TEST_CASES = [
  {
    description: 'parses Active status correctly',
    given: { statusText: 'Active' },
    expect: { status: DeclaredSquarespaceDomainRegistrationStatus.ACTIVE },
  },
  {
    description: 'parses Expired status correctly',
    given: { statusText: 'Expired' },
    expect: { status: DeclaredSquarespaceDomainRegistrationStatus.EXPIRED },
  },
];

describe('parseStatus', () => {
  TEST_CASES.forEach(c =>
    test(c.description, () => {
      const result = parseStatus(c.given.statusText);
      expect(result).toEqual(c.expect.status);
    })
  );
});
```

### pattern 3: idempotency verification

```ts
when('applying the same plan twice', () => {
  then('second apply succeeds without errors', async () => {
    // first apply
    await execSync(`npx declastruct apply --plan ${planFile}`);

    // second apply (should be idempotent)
    expect(() =>
      execSync(`npx declastruct apply --plan ${planFile}`)
    ).not.toThrow();
  });

  then('re-plan shows all KEEP', async () => {
    const plan = await generatePlan();
    const changes = plan.changes.filter(c => c.action !== 'KEEP');
    expect(changes).toHaveLength(0);
  });
});
```
