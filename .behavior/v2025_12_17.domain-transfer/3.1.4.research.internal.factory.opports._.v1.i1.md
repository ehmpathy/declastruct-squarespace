# research: factory opportunities

## the constraint

**the bottleneck**: playwright browser automation speed

each operation requires:
1. navigate to page (~1-2 sec)
2. wait for elements (~0.5-1 sec)
3. interact (~0.5 sec)
4. wait for confirmation (~0.5-1 sec)

total: ~2-5 sec per operation

for 300 domains: ~15-25 min just for enumeration, ~1-3 hours for full transfer prep

**why not parallelize**: rate limit risk, bot detection risk, ui automation state complexity

**accepted constraint**: sequential is safer; cache mitigates enumeration cost

---

## flow and queues

**queues in verification**:
1. playwright browser launch (~2 sec cold start)
2. authentication flow (~5-10 sec per login)
3. each scrape operation (~2-5 sec)

**wait outside control**: squarespace page load times

**wip limit**: one domain at a time (sequential)

---

## waste

**handoffs**: none (single tool, single process)

**repeated setup** (current state):
- browser launch → in-memory cache (lost between runs)
- authentication → fresh login each test run (~5-10 sec + bot detection risk)
- domain enumeration → on-disk cache (persists)

**repeated setup** (with factory upgrade):
- browser launch → connect to persistent browser server
- authentication → restore session from disk (skip login if valid)
- domain enumeration → on-disk cache (persists)

**partial work idle**: none (each operation is atomic)

**manual error-prone**: none (fully automated)

## opportunities identified

### opportunity 1: session persistence (HIGH LEVERAGE) — IMPLEMENTED

**the problem**: every test run requires fresh login

previously:
- browser cache is in-memory (`SimpleInMemoryCache`)
- lost between test runs / process restarts
- each test run triggers login flow (~5-10 sec)
- frequent logins increase bot detection risk
- frequent logins increase rate limit risk

**solution implemented**:
- playwright `storageState` persists cookies/session to disk
- session restored when browser context is created
- login only when session state is absent or expired (health check fails)

**implementation**:
- `ContextSquarespaceAgentOptions.session.storageStatePath` config option
- `genLoggedInBrowser.ts` restores storageState from disk if file extant
- `genLoggedInBrowser.ts` checks session health before login
- `genLoggedInBrowser.ts` saves storageState after successful login
- test infra supports via `SESSION_STORAGE_PATH` env var

**best practices applied** (per playwright docs):
- storageState captures cookies, localStorage, indexedDB
- session health check validates before reuse (token expiration)
- separate storageState file per account supported

**sources**:
- [Playwright Authentication](https://playwright.dev/docs/auth)
- [BrowserStack storageState Guide](https://www.browserstack.com/guide/playwright-storage-state)

**effort**: medium — COMPLETE
**impact**: high (eliminates repeated logins, reduces bot detection risk)
**scope**: factory-wide (benefits all test runs)

### opportunity 2: persistent browser server — IMPLEMENTED

**the problem**: browser launches fresh each run

previously:
- browser launched per test run (~2 sec cold start)
- `extantBrowserWSEndpoint` option exists but rarely used

**solution implemented**:
- `BROWSER_WS_ENDPOINT` env var connects to persistent browser server
- combined with session persistence, eliminates both browser launch AND login
- documented in factory blueprint

**usage**:
```sh
# terminal 1: launch persistent browser server
npx playwright launch chromium --port 9222

# terminal 2: run tests with browser + session reuse
BROWSER_WS_ENDPOINT=ws://localhost:9222/devtools/browser/xxx \
SESSION_STORAGE_PATH=.cache/squarespace-session.json \
npm run test:integration
```

**best practices applied** (per playwright docs):
- `chromium.connectOverCDP()` connects to persistent browser
- browser context created within persistent browser (not new launch)
- combined with storageState for maximal reuse

**sources**:
- [BrowserStack Persistent Context](https://www.browserstack.com/guide/playwright-persistent-context)

**effort**: low — COMPLETE
**impact**: medium (faster test iteration)
**scope**: developer workflow (factory-wide)

### opportunity 3: eager cache preload (DEFERRED)

currently:
- cache populated on first access
- cold cache = full scrape

opportunity:
- batch-populate cache on provider init
- preloaded cache reduces plan time from minutes to seconds

**effort**: low (add optional eager load)
**impact**: moderate (saves ~1-5 min on cold plan)
**scope**: local to this behavior (not factory-wide)

**decision**: defer — session persistence has higher leverage
