## tools already configured

### jest (test runner)

**package**: jest v30.2.0

**configuration files:**
- `jest.config.ts` — shared base config
- `jest.unit.config.ts` — unit tests (`*.test.ts`)
- `jest.integration.config.ts` — integration tests (`*.integration.test.ts`)
- `jest.acceptance.config.ts` — acceptance tests (`*.acceptance.test.ts`)

**npm scripts:**
```json
"test:unit": "jest -c ./jest.unit.config.ts --forceExit --verbose --passWithNoTests $([ -z $THOROUGH ] && echo '--changedSince=main')",
"test:integration": "jest -c ./jest.integration.config.ts --forceExit --verbose --passWithNoTests $([ -z $THOROUGH ] && echo '--changedSince=main')",
"test:acceptance:locally": "npm run build && LOCALLY=true jest -c ./jest.acceptance.config.ts --forceExit --verbose --runInBand --passWithNoTests",
"test:integration:watch": "jest -c ./jest.integration.config.ts --watch"
```

**rapid feedback feature:** `--changedSince=main` runs only tests for changed files.

### test-fns (bdd structure)

**package**: test-fns v1.7.2

**citation [1]**: test-fns npm
> "write usecase-driven tests for simpler, safer code."
> source: https://www.npmjs.com/package/test-fns

**patterns used:**
- `given()` for scenario setup
- `when()` for action context
- `then()` for assertions
- `useBeforeAll()` for async fixture setup

**example from codebase:**
```ts
describe('scrapeDomainsList', () => {
  given('a logged-in browser session', () => {
    const scene = useBeforeAll(async () => {
      const context = getSampleSquarespaceContext();
      const page = await getNewLoggedInBrowserPage(context.agentOptions);
      return { context, page };
    });

    when('scrape the domains list page', () => {
      const result = useBeforeAll(async () => {
        const domains = await scrapeDomainsList({ page: scene.page });
        return { domains };
      });

      then('returns an array of domains', async () => {
        expect(Array.isArray(result.domains)).toBe(true);
      });
    });
  });
});
```

### playwright (browser automation)

**package**: playwright v1.49.1

## rapid feedback features

### watch mode

```sh
npm run test:integration:watch
```

runs tests in watch mode, re-runs on file change.

### changed-only mode

```sh
npm run test:integration
# runs only tests for files changed since main branch
```

`--changedSince=main` flag enables incremental test runs.

### full run mode

```sh
THOROUGH=true npm run test:integration
# runs all tests regardless of changes
```

`THOROUGH=true` bypasses `--changedSince=main`.

### trace capture

```sh
npm run test:trace
# captures playwright traces for debug
```

`PLAYWRIGHT_TRACE=on` enables trace capture for failed tests.

### cache clear

```sh
npm run cache:clear
# clears remote state cache
```

clears `.cache/squarespace` to force fresh scrape.

## persistent browser for faster iteration

development workflow uses persistent browser to avoid cold start.

**terminal 1: launch persistent browser**
```sh
npx playwright launch chromium --port 9222
```

**terminal 2: run tests with browser reuse**
```sh
BROWSER_WS_ENDPOINT=ws://localhost:9222/devtools/browser/xxx npm run test:integration
```

**session reuse:**
```sh
SESSION_STORAGE_PATH=.cache/squarespace-session.json npm run test:integration
```

- session restored from disk (no login if valid)
- minimizes login frequency
- reduces bot detection risk

## gaps and solutions

### gap 1: no visual diff for UI changes

**problem:** squarespace UI changes could break selectors without immediate visibility.

**solution:** playwright trace capture + trace viewer.

```sh
npm run test:trace -- scrapeDomainsList.integration.test.ts
npx playwright show-trace trace.zip
```

### gap 2: long test cycles for full batch

**problem:** 300 domains = hours of test time.

**solution:** integration tests use single domain; acceptance tests verify workflow; batch is production operation.

### gap 3: flaky tests from network variance

**problem:** network latency and squarespace response time vary.

**solution:** playwright auto-wait for selectors; explicit timeouts where needed.

## comparison: playwright vs puppeteer

| aspect | playwright | puppeteer |
|--------|------------|-----------|
| browser support | chromium, firefox, webkit | chromium only |
| auto-wait | yes | manual |
| api style | async/await | async/await |
| stealth plugin | via playwright-extra | via puppeteer-extra |
| microsoft support | yes | google |
| context isolation | yes | yes |

**decision:** playwright chosen for auto-wait and multi-browser support.
