## declastruct-stripe-sdk template

**source**: https://github.com/ehmpathy/declastruct-stripe-sdk
**reference**: `.agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/readable.persistence/rule.prefer.declastruct.[demo].md`

### pattern 1: domain entity declaration

domain entities declare primary and unique keys explicitly.

```ts
export interface DeclaredStripeCustomer {
  id?: string;                           // primary key (artificial)
  email: string;                         // unique key (natural)
  name: null | string;
  description: null | string;
  phone: null | string;
  metadata: null | Record<string, string>;
}

export class DeclaredStripeCustomer
  extends DomainEntity<DeclaredStripeCustomer>
  implements DeclaredStripeCustomer
{
  public static primary = ['id'] as const;
  public static unique = ['email'] as const;
  public static nested = { metadata: DomainLiteral };
}
```

**key patterns:**
- `primary` = artificial key for foreign references
- `unique` = natural key for idempotency
- nullable attributes explicit with `null | T`
- no `undefined` attributes (except database-generated metadata like `id`)

**relation to wish:**
- `DeclaredSquarespaceDomainRegistration` uses `name` as unique key (natural domain identifier)
- `DeclaredSquarespaceDomainDnsRecord` uses composite `[domain, type, host]` as unique key
- `DeclaredSquarespaceDomainTransferRequest` uses `domain` as unique key

### pattern 2: cast from sdk shape to declared shape

translate from external sdk representation to domain object.

```ts
export const castToDeclaredStripeCustomer = (
  input: Stripe.Customer,
): HasMetadata<DeclaredStripeCustomer> => {
  return new DeclaredStripeCustomer({
    id: input.id,
    email: input.email ?? UnexpectedCodePathError.throw('no email found'),
    description: input.description ?? null,
    name: input.name ?? null,
    phone: input.phone ?? null,
    metadata: (() => {
      const obj = input.metadata ? omit(input.metadata, ['exid']) : {};
      if (Object.keys(obj).length === 0) return null;
      return obj;
    })(),
  }) as HasMetadata<DeclaredStripeCustomer>;
}
```

**key patterns:**
- explicit null fallback for optional sdk fields via `??`
- fail-fast on required fields via `UnexpectedCodePathError.throw`
- cast to `HasMetadata<T>` to indicate object has been hydrated

**relation to wish:**
- `castIntoDeclaredSquarespaceDomainRegistration` casts scraped data to domain object
- `castIntoDeclaredSquarespaceDomainDnsRecord` casts dns records
- `castIntoDeclaredSquarespaceDomainTransferRequest` casts transfer requests

### pattern 3: get operations with by.primary, by.unique, by.ref

support multiple lookup strategies.

```ts
export const getCustomer = async (
  input: {
    by: PickOne<{
      primary: RefByPrimary<typeof DeclaredStripeCustomer>;
      unique: RefByUnique<typeof DeclaredStripeCustomer>;
      ref: Ref<typeof DeclaredStripeCustomer>;
    }>;
  },
  context: StripeApiContext & VisualogicContext,
): Promise<HasMetadata<DeclaredStripeCustomer> | null> => {
  // handle by ref (dispatch to primary or unique)
  if (input.by.ref)
    return isUniqueKeyRef({ of: DeclaredStripeCustomer })(input.by.ref)
      ? getCustomer({ by: { unique: input.by.ref } }, context)
      : getCustomer({ by: { primary: input.by.ref } }, context);

  // handle by primary
  if (input.by.primary) {
    const customer = await context.stripe.customers.retrieve(input.by.primary.id);
    if (customer.deleted) return null;
    return castToDeclaredStripeCustomer(customer);
  }

  // handle by unique
  if (input.by.unique) {
    const { data: [customer] } = await context.stripe.customers.list({
      email: input.by.unique.email,
    });
    if (!customer) return null;
    return castToDeclaredStripeCustomer(customer);
  }
};
```

**key patterns:**
- `PickOne<>` enforces exactly one lookup strategy
- `RefByPrimary`, `RefByUnique`, `Ref` for typed references
- `isUniqueKeyRef` to dispatch ref to correct lookup
- returns `null` for not found (not error)

**relation to wish:**
- squarespace has no api, so lookup is via scrape
- `getOneDomain` uses `by.unique` (domain name)
- `getOneDnsRecord` uses composite unique key
- `getOneTransferRequest` uses domain reference

### pattern 4: set operations with findsert and upsert

idempotent mutation semantics.

```ts
export const setCustomer = withLogTrail(
  async (
    input: PickOne<{
      findsert: DeclaredStripeCustomer;
      upsert: DeclaredStripeCustomer;
    }>,
    context: StripeApiContext & VisualogicContext,
  ): Promise<HasMetadata<DeclaredStripeCustomer>> => {
    // lookup by unique key
    const customerFound = await getCustomer(
      { by: { unique: { email: input.findsert?.email ?? input.upsert?.email } } },
      context,
    );

    // findsert: return found if extant
    if (customerFound && input.findsert) return customerFound;

    // upsert: update if extant
    if (customerFound && input.upsert) {
      return castToDeclaredStripeCustomer(
        await context.stripe.customers.update(customerFound.id, { ... }),
      );
    }

    // create if not extant
    const customerCreated = await context.stripe.customers.create({ ... });
    return castToDeclaredStripeCustomer(customerCreated);
  },
);
```

**key patterns:**
- `PickOne<{ findsert, upsert }>` enforces exactly one mutation mode
- `findsert` = find extant or insert (preserves extant attributes)
- `upsert` = update extant or insert (overwrites extant attributes)
- lookup by unique key first, then decide create vs update
- idempotency key for create operations

**relation to wish:**
- `setDomain` uses upsert for `isLocked`, `dnssecEnabled` attributes
- `setTransferRequest` uses findsert (find extant request or create new)
- both patterns ensure safe re-run of batch operations

### pattern 5: integration tests with real resources

tests verify shape and idempotency against live service.

```ts
describe('setCustomer', () => {
  given('a .findsert', () => {
    when('the customer does not exist yet', () => {
      const email = `test.${getUuid()}@example.com`;
      then('we should create it', async () => {
        const customer = await setCustomer({ findsert: { email, name: 'test' } }, context);
        expect(customer.id).toContain('cus_');
      });
    });

    when('the customer already exists', () => {
      const customerBefore = useBeforeAll(async () =>
        await setCustomer({ findsert: { email, name: 'original' } }, context),
      );
      then('it should not update the customers attributes', async () => {
        const customerAfter = await setCustomer({ findsert: { email, name: 'new' } }, context);
        expect(customerAfter.name).toEqual(customerBefore.name);
      });
    });
  });
});
```

**key patterns:**
- given/when/then structure via test-fns
- `useBeforeAll` for fixture setup
- verify idempotent behavior (findsert preserves, upsert overwrites)
- test both create and find paths

**relation to wish:**
- integration tests for scrapers verify shape
- acceptance tests verify full provider workflow
- idempotency tests ensure safe batch re-runs

## declastruct-aws template

**source**: declastruct-aws (this repo's prior implementation)

### pattern: provider composition

provider bundles context and daos.

```ts
export const getDeclastructSquarespaceProvider = (
  config: ProviderConfig,
): DeclastructProvider => {
  const agentOptions = getSquarespaceAgentOptions(config);
  const context = { agentOptions };

  return {
    context,
    daos: {
      DeclaredSquarespaceDomainRegistration: {
        get: {
          all: (input) => getAllDomains(input, context),
          one: { byUnique: (input) => getOneDomain(input, context) },
        },
        set: {
          upsert: (input) => setDomain(input, context),
        },
      },
      // ... other daos
    },
  };
};
```

**key patterns:**
- context baked into daos (no explicit context pass per call)
- daos structured as `get.all`, `get.one.byUnique`, `set.upsert`, `set.findsert`
- provider is single entry point for all operations

**relation to wish:**
- `getDeclastructSquarespaceProvider` follows this exact pattern
- context includes playwright browser, cache, rate limiter
- daos wrap domain operations with baked context

## summary: template patterns applied

| pattern | stripe template | squarespace application |
|---------|----------------|------------------------|
| domain entity with primary/unique | DeclaredStripeCustomer | DeclaredSquarespaceDomainRegistration |
| cast from sdk shape | castToDeclaredStripeCustomer | castIntoDeclaredSquarespaceDomainRegistration |
| get by primary/unique/ref | getCustomer | getOneDomain (unique only, no api) |
| set findsert/upsert | setCustomer | setDomain (upsert), setTransferRequest (findsert) |
| provider composition | getStripe | getDeclastructSquarespaceProvider |
| integration tests | real stripe api | real squarespace ui via playwright |

**key difference:**
- stripe has api â†’ direct sdk calls
- squarespace has no api â†’ playwright ui automation

the patterns remain identical; only the access layer differs.
