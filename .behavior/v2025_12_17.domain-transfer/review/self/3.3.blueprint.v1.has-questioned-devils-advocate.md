## devil's advocate review

### criticism 1: "ui automation is the wrong tool entirely"

**the skeptic says:**
"you should have negotiated api access with squarespace. enterprise accounts get special treatment. you're constructing a fragile house of cards instead of addressing the root problem."

**my response:**
- contacted squarespace support: no api extant, no plans to build one
- enterprise accounts don't get api access — just dedicated support
- migration is time-sensitive (domain expirations, renewal costs)
- ui automation is the only path forward

**verdict:** no better alternative extant. the skeptic's suggestion requires squarespace to change, which is outside our control.

### criticism 2: "this is over-engineered for a one-time migration"

**the skeptic says:**
"300 domains is a one-time transfer. just hire a virtual assistant to click through the ui manually. this codebase will be abandoned after one use."

**my response:**
- 50+ hours of manual work vs 10 hours to build and 3 hours to run
- declarative approach enables verification (plan before apply)
- idempotency enables safe retries on failures
- codebase can be reused if user acquires more domains later
- skills learned transfer to other ui-only platforms

**verdict:** the investment is justified. automation quality exceeds manual work quality.

### criticism 3: "selector fragility makes this unmaintainable"

**the skeptic says:**
"squarespace will update their ui. your selectors will break. you'll spend more time on maintenance than you saved. this is technical debt disguised as automation."

**my response:**
- selectors centralized in 4 files — clear maintenance points
- fallback pattern (data-test, class) reduces single-point failures
- integration tests detect breakage immediately
- maintenance is bounded: fix selectors, re-run tests
- one-time migration means maintenance window is limited

**verdict:** fragility is real but contained. limited maintenance window (migration period) bounds the risk.

### criticism 4: "no playwright-stealth means you'll get blocked"

**the skeptic says:**
"you documented that playwright-stealth is absent. you know bot detection is a risk. why ship without it? you're positioning the user for failure at domain 147."

**my response:**
- valid criticism
- **investigation findings:**
  - vision claims "uses playwright-stealth" but package.json has NO stealth dependency
  - `playwright-stealth` (npm 0.0.1) is just a placeholder — not a real package
  - actual browser code in `genLoggedInBrowser.ts` lines 33-37 only sets userAgent
  - no actual stealth mechanism is implemented
- current mitigation (userAgent + 500ms delay) is insufficient against modern bot detection

**technical fix required:**
1. add `playwright-extra` (4.3.6) — plugin framework for playwright
2. add `puppeteer-extra-plugin-stealth` (2.11.2) — works with both puppeteer-extra AND playwright-extra
3. modify `genLoggedInBrowser.ts`:
```ts
import { chromium } from 'playwright-extra';
import StealthPlugin from 'puppeteer-extra-plugin-stealth';

chromium.use(StealthPlugin());
// then launch as normal
```

**verdict:** the skeptic is right. this is a **critical gap** — vision claims stealth but implementation has none.

**action:** dependencies require human approval; documented as pre-production blocker.

### criticism 5: "24-hour cache will cause stale data disasters"

**the skeptic says:**
"user plans in the am, applies in pm. between those times, someone manually changes domain state. cache doesn't know. apply fails silently or worse — makes wrong decisions."

**my response:**
- mutations invalidate cache
- user can clear cache directory manually
- plan/apply workflow is meant to be close in time
- for one-time migration, user controls the time

**verdict:** the skeptic has a point for general use, but workflow assumptions mitigate for migration use case. could add --force-refresh flag as enhancement.

### criticism 6: "why not just use puppeteer?"

**the skeptic says:**
"puppeteer is more mature. larger community. more examples. you chose playwright just because it's newer. that's not a technical reason."

**my response:**
- playwright has better typescript support
- playwright handles multiple browsers (chromium, firefox, webkit)
- playwright-stealth plugin exists for puppeteer too
- choice is pragmatic preference, not critical

**verdict:** either would work. not a fundamental flaw.

### criticism 7: "you didn't test with real batch size"

**the skeptic says:**
"you tested with a few domains. 300 domains is 100x larger. memory usage, session timeouts, rate limits — all unknown at scale. you assume it works."

**my response:**
- valid criticism
- integration tests cover single operations
- no scale test with 100+ domains
- should run pilot batch (e.g., 50 domains) before full migration

**verdict:** the skeptic is right. scale validation is needed.

**action:** document pilot batch as pre-production requirement.

## blind spots exposed

### blind spot 1: optimism about squarespace stability

**what we want to believe:**
squarespace won't change their ui mid-migration.

**reality:**
squarespace ships updates constantly. migration takes days/weeks. ui could change mid-batch.

**mitigation:**
plan for selector updates. keep integration tests green. run migration in focused bursts.

### blind spot 2: trust in bot detection evasion

**what we want to believe:**
500ms delay + userAgent is enough to avoid detection.

**reality:**
- no evidence this is true — just an assumption
- worse: vision claims "uses playwright-stealth" but **it's not in the codebase**
- actual implementation only sets userAgent (line 34-35 of genLoggedInBrowser.ts)
- no actual stealth fingerprint masking is present

**mitigation:**
- added `.todo` in genLoggedInBrowser.ts documenting the fix path
- fix requires: playwright-extra + puppeteer-extra-plugin-stealth (needs human approval)

### blind spot 3: assumption about registrar uniformity

**what we want to believe:**
all registrar backends use the same ui flow.

**reality:**
only tested with one registrar type. tucows and key_systems are untested.

**mitigation:**
test with domains from each registrar before full migration.
