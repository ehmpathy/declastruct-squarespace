## issues found

### issue 1: hardcoded `waitForTimeout(3000)` in requestTransferCode.ts

**code found:**
```ts
// requestTransferCode.ts:90
await page.waitForTimeout(3000);
```

**5 whys:**
1. why hardcoded sleep? → to wait for success message after button click
2. why not wait for selector? → unknown what success element appears
3. why unknown? → squarespace ui wasn't fully explored during implementation
4. why not explored? → time pressure to get it working
5. why is this bad? → race condition: sometimes 3s is too short, sometimes too long

**verdict:** code smell. should be `waitForSelector` on success element with timeout fallback. document as known fragility.

### issue 2: no retry logic when selectors fail

**code found:**
```ts
// requestTransferCode.ts:44-48
let requestButton = await page.$(transferSelectors.requestCodeButton);
if (!requestButton) {
  requestButton = await page.$(transfersListSelectors.generateAuthCodeButton);
}
```

**5 whys:**
1. why check two selectors? → squarespace has different ui variants
2. why not retry with wait? → assumed one selector would immediately work
3. why assume that? → worked during test implementation
4. why might it fail? → page load race, ui update, spa navigation delay
5. why is this bad? → intermittent failures on slow networks

**verdict:** fragile selector lookup. should wait for either selector with timeout.

### issue 3: session expiry not proactively detected

**code found:**
```ts
// getSquarespaceAgentOptions.ts:42-44
const browserCache = createInMemoryCache<LoggedInBrowser>({
  expiration: { hours: 1 },
});
```

**5 whys:**
1. why 1-hour browser cache? → assumed sessions last long enough
2. why assumed? → no evidence squarespace session duration
3. what happens at expiry? → cache clears, next operation creates fresh browser
4. what about mid-operation expiry? → handleReauthentication catches modal
5. why might this fail? → if session expires without modal (redirect to login), scrape fails

**verdict:** reactive reauth exists but proactive session check is absent. document as risk.

### issue 4: operation fails after page invalidation, no retry

**code found:**

} catch (error) {

## decisions validated via 5 whys

### decision: findsert-only for transfer requests

1. why findsert, not upsert? → transfer requests are immutable
2. why immutable? → squarespace doesn't allow modify/cancel via ui
3. why expose upsert: null? → explicit communication of intent
4. why does explicit null matter? → declastruct introspects dao capabilities
5. why is this correct? → matches domain reality: transfers can't be modified

**verdict:** valid. findsert semantics match domain constraints.

### decision: 500ms sequential delay via bottleneck

1. why 500ms? → mimics human pace
2. why not configurable? → one-time migration doesn't need tuning
3. why sequential? → parallel is bot signature
4. why is bottleneck used? → provides maxConcurrent and minTime
5. why external library? → well-tested rate limit implementation

**verdict:** valid. appropriate for batch migration use case.

### decision: composite key (domain + type + host) for dns records

1. why composite? → squarespace doesn't expose stable ids
2. why these fields? → dns semantics: record = (name, type, host)
3. why not generate id? → breaks idempotency on re-scrape
4. why is idempotency critical? → declastruct compares desired vs actual
5. why does this work? → matches how dns actually works

**verdict:** valid. reflects dns domain model.
