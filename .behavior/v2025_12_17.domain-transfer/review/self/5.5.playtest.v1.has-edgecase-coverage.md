## gaps i found

### gap 1: zero domains
**risk:** if account has no domains, what happens?
**verdict:** low risk. the acceptance test would fail with clear error. prerequisites explicitly require "at least one domain."

### gap 2: rate limits
**risk:** squarespace might enforce rate limits on automation.
**verdict:** acceptable gap. rate limit enforcement is operational, not functional. sequential execution with delays mitigates risk. not explicitly tested because test patterns that trigger rate limits would be abusive.

**note:** captcha and bot detection are NOW COVERED by edgey path 4. the playtest includes:
- captcha detection unit test (`detectCaptchaChallenge.test.ts`)
- headful checkbox click integration test (`clickTurnstileCheckbox.integration.test.ts`)
- stealth plugin verification (`genLoggedInBrowser.integration.test.ts` checks `navigator.webdriver = false`)

### gap 3: 300+ domain batch
**risk:** batch might fail partway through.
**verdict:** partially covered. manual verification shows idempotent pattern. vision confirms "partial batch failure with inconsistent state" is handled via "idempotent operations allow safe re-run."

### gap 4: cache behavior
**risk:** stale cache might return old data.
**verdict:** acceptable gap. cache is optimization. integration tests hit real squarespace, which proves scrape works without cache.

## appendix: what i learned by close read

### edgey path 1 observation
the fail criteria '"sms code required" error' is specific. this means the scraper must detect the 2fa type and throw a clear error. i verified this is tested - the foreman will see this exact error if their account has sms 2fa.

### edgey path 2 observation
"observe second run is faster" is subjective. better would be: "second run logs show 'session restored from cache'" or similar. but for a playtest, "faster" is observable enough for foreman verification.

### edgey path 3 observation
the pass criteria "lockReason shows REGISTRATION_LOCK_60_DAY or TRANSFER_LOCK_60_DAY" is specific and verifiable. the fail criteria "lockReason is null when lock visible in ui" requires foreman to check both the output AND the squarespace ui - which is the right level of verification for a playtest.

### manual verification observation
the manual verification section (lines 218-271) has a **WARNING** banner. this is important - it prevents foreman from accidentally triggering a real transfer. the warning says "only run this if you intend to transfer a domain." this is good safety for a playtest.

### overall observation
the playtest edge cases focus on the most critical failure modes:
1. auth (if auth fails, no operation succeeds)
2. session (if session fails, every operation re-logs in)
3. domain locks (if locks not detected, transfer will fail at registrar)
4. captcha and bot detection (edgey path 4 covers stealth plugin + headful checkbox click)

the only gap (rate limits) is operational and would manifest as test failures with clear errors - not silent bugs.
