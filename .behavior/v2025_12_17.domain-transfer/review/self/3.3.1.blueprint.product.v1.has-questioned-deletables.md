## component-by-component analysis

### domain objects

#### DeclaredSquarespaceDomainRegistration

**can it be removed?** no — core entity for domain transfer workflow

**would we add it back?** yes — represents the primary resource

**verdict:** keep

#### DeclaredSquarespaceDomainDnsRecord

**can it be removed?** maybe

**analysis:** the wish says "get operations to see the dns of those domains". this is read-only. do we need a full domain object for dns records, or could we just return raw scrape data?

**counterargument:** dns records need to be preserved before transfer. a typed domain object ensures correct shape for export/reimport.

**would we add it back?** yes — typed shape helps preserve dns config

**verdict:** keep, but blueprint already removes set/del operations (yagni)

#### DeclaredSquarespaceDomainTransferRequest

**can it be removed?** no — tracks transfer code requests

**analysis:** needed for idempotent findsert semantics. without it, we cannot track "already requested" state.

**would we add it back?** yes

**verdict:** keep

## simplification opportunities

### opportunity 1: merge get operations

**could we merge `getAllDomains` and `getOneDomain`?**

no — different codepaths. `getAll` scrapes list page. `getOne` could filter from cached list. separation is cleaner.

**verdict:** keep separate

### opportunity 2: eliminate cache layer

**could we remove remote state cache?**

no — without cache, every plan operation re-scrapes all domains (~5 min for 300 domains). cache is essential for usability.

**verdict:** keep

### opportunity 3: simpler transfer request

**could transfer request be a simple boolean on DomainRegistration?**

no — transfer requests have their own lifecycle (requested, queued, approved, rejected). separate entity is correct.

**verdict:** keep separate
