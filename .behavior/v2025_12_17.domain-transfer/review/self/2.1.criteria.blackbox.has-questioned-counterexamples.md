## usecase-by-usecase analysis

### usecase.1 — enumerate domains

**counterexample: 10,000 domains in account**

the criterion says "returns list of all domain registrations". what if enumeration takes 5 minutes and times out?

- pass criterion, fail vision? no — timeout fails the criterion itself
- fail criterion, fulfill vision? yes — first 500 with pagination would let user proceed incrementally

**verdict**: the vision targets 300+ domains. 10k is hypothetical. criterion stands.

**counterexample: domain in inbound transfer state**

what if a domain is mid-transfer INTO squarespace from another registrar?

- pass criterion, fail vision? yes — domain appears in list, but can't transfer OUT

**verdict**: usecase.3 lockReason surfaces this. no gap.

### usecase.2 — check dns records

**counterexample: squarespace system records**

what if enumeration returns squarespace-internal records (www → sqsp host)?

- pass criterion, fail vision? possible — user gets records they didn't create

**verdict**: extra records > absent records. user can filter. criterion stands.

**counterexample: malformed TTL value**

what if TTL is 0 or negative?

- pass criterion, fail vision? no — data fidelity is the goal, not interpretation

**verdict**: criterion stands.

### usecase.3 — unlock domain for transfer

**counterexample: unlocked but blocked by verification**

### usecase.5 — request transfer authorization code

**counterexample: email bounces**

transfer request succeeds but email never arrives (bounce, spam filter).

- pass criterion, fail vision? yes — user has entity but no usable code

**verdict**: email delivery is out of scope. vision documents this as "awkward". criterion tests the request, not delivery.

**counterexample: expired transfer code**

user requested code 3 weeks ago. findsert returns extant request. but code expired.

- pass criterion, fail vision? **yes** — user thinks they have a code, but it's invalid

**analysis**: transfer codes typically expire after 7-14 days. the findsert idempotency breaks down when the extant request is stale.

**verdict**: **gap found**. need scenario for expired code re-request.

**action taken**: added scenario for expired transfer code (14+ days old triggers new request).

## gaps found and fixed

three gaps identified in this counterexample review:

### gap 1: awaited verification block (usecase.3)

**problem**: domain can be unlocked but still untransferable due to verification.

**fix**: added to criteria:
```
given('a domain unlocked but with awaited verification')
  when('user requests transfer code')
    then('returns error: domain has awaited verification')
      sothat('user knows additional steps are required')
```

### gap 2: expired transfer codes (usecase.5)

**problem**: findsert returns stale request with expired code.

**fix**: added to criteria:
```
given('a transfer code requested more than 14 days ago')
  when('user requests transfer code again (findsert)')
    then('creates new transfer request')
      sothat('user receives fresh valid code')
    then('returns new transfer request entity')
      sothat('user knows a new code was sent')
```

### gap 3: session refresh failure (usecase.7)

**problem**: criterion assumes refresh always succeeds.

**fix**: added to criteria:
```
given('session refresh fails mid-batch')
  when('next operation is attempted')
    then('returns error: session refresh failed with reason')
      sothat('user knows re-authentication is required')
    then('batch halts at current position')
      sothat('user can fix credentials and resume')
```

## second pass: deeper analysis

after the initial review, i re-examined with fresh perspective.

### error message consistency

the criteria use illustrative error messages ("returns error: X"). these are semantic, not literal. implementation may vary text. blackbox tests verify intent, not exact text.

**verdict**: appropriate for blackbox. no change needed.

### internationalized domain names (IDN)

criteria don't mention IDN (e.g., 日本語.com). squarespace UI displays unicode. selectors that work with text content should handle IDN.

**verdict**: implementation concern, not criterion concern. criterion tests enumeration — any valid domain should appear.

### concurrent modification

what if another admin modifies domain state mid-batch? the vision says "one account at a time" and "sequential". concurrent modification is user error. idempotent semantics handle it — worst case, re-run.

**verdict**: no change needed.
