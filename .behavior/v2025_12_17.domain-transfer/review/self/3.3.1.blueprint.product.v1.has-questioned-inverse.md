## architectural states to avoid

### 1. parallel operations

**what to avoid:** concurrent playwright operations on squarespace

**why:** squarespace rate limits and bot detection would trigger

**current implementation:** bottleneck with `maxConcurrent: 1` + delays

**verification:** checked `withNewLoggedInBrowserPage.ts` - all operations go through bottleneck

**verdict:** correctly avoided

---

### 2. shared mutable state

**what to avoid:** global variables, mutated configs, shared browser state between calls

**why:** race conditions, unpredictable behavior, test interference

**current implementation:**
- context passed via dependency injection
- browser cache keyed by account + credentials
- no global mutable state

**verification:** checked for `let` at module level - none found in domain operations

**verdict:** correctly avoided

---

### 3. barrel exports

**what to avoid:** `export * from './file'` patterns in index.ts

**why:** creates circular import risks, slower builds, harder to trace imports

**current implementation:** no barrel exports in daos or domain.operations

**verification:** no `index.ts` barrel files found in implementation directories

**verdict:** correctly avoided (per repo rule.forbid.barrel-exports)

---

### 4. untyped bag refs

**what to avoid:** `{ name: domain.name }` instead of `RefByUnique.as<>()`

**why:** loses type safety, intellisense, and correct property pick

**current implementation:** all domain refs use `RefByUnique<typeof Entity>` or `RefByUnique.as<>()`

**verification:** checked cast functions and DAOs - all use RefByUnique pattern

**verdict:** correctly avoided (per repo rule.forbid.dobj.bagrefs)

## dependencies that would create problems

### 1. puppeteer instead of playwright

**what to avoid:** use puppeteer or other browser automation

**why:** playwright is modern, has better API, better maintained

**current implementation:** uses `playwright` exclusively

**verdict:** correct choice

---

### 2. mock in tests

**what to avoid:** `jest.mock()` and fake implementations

**why:** mocks lie, drift from reality, create false confidence

**current implementation:** integration tests use real squarespace, unit tests use dependency injection

**verification (2026-03-12):** grep found 1 mock in `answerCaptchaChallenge.test.ts:111`:
- mocks `waitForHumanSignal` — human interaction (stdin ENTER)
- acceptable: can't have real human press ENTER in automated test

**verdict:** correctly avoided for business logic. exception for human interaction mock is acceptable.

---

### 3. direct filesystem in operations

**what to avoid:** `fs.readFile`, `fs.writeFile` in domain.operations

**why:** breaks unit testability, couples to filesystem

**current implementation:** cache abstraction via `SimpleOnDiskCache` from external package

**verdict:** correctly abstracted

## tech debt creation risks

### 1. magic timeouts

**what to avoid:** `waitForTimeout(3000)` without explanation

**current status:** found 4 instances:
- `toggleDomainLock.ts:66` - 2000ms
- `requestTransferCode.ts:92` - 3000ms with `.todo` comment
- `scrollToLoadAllDomains.ts:44` - 1000ms
- `toggleDnssec.ts:74` - 2000ms

**assessment:** these are acknowledged fragile patterns. the `.todo` comment in requestTransferCode.ts shows awareness.

**verdict:** known tech debt, documented for future fix. not a blocker.

---

### 2. hardcoded selectors

**what to avoid:** selectors without fallbacks that break on UI changes

**current status:** selectors use fallback patterns:
```ts
'[data-test="request-code"], button[aria-label*="Transfer"], button:has-text("Transfer")'
```

**assessment:** fallbacks reduce fragility

**verdict:** acceptable approach

---

### 3. implicit nullable attributes

**what to avoid:** `field?: string` instead of `field: string | null`

**current status:** domain objects use explicit `| null` for nullable fields

**verification:** checked DeclaredSquarespaceDomainRegistration - uses `lockReason: DeclaredSquarespaceDomainLockReason | null`

**verdict:** correctly avoided (per repo rule.forbid.undefined-attributes)

## anti-patterns to steer clear of

### 1. error swallow

**what to avoid:** try/catch blocks that hide errors

**current status:** found try/catch in:
- `withNewLoggedInBrowserPage.ts` - rethrows after cleanup
- `checkSessionHealth.ts` - returns failure status, doesn't hide

**assessment:** both patterns are correct - they handle errors properly

**verdict:** correctly avoided

---

### 2. create/insert verbs

**what to avoid:** non-idempotent mutation verbs

**current status:** uses `findsert` and `upsert` exclusively

**verification:** no `create*` or `insert*` operation names found

**verdict:** correctly avoided (per repo rule.forbid.nonidempotent-mutations)

### 3. positional arguments

**what to avoid:** `fn(a, b, c)` instead of `fn({ a, b, c })`

**current status:** all operations use `(input, context)` pattern

**verdict:** correctly avoided (per repo rule.require.input-context-pattern)

## why each avoidance holds

### architectural states — why they hold

**parallel operations (avoided):**
- bottleneck with `maxConcurrent: 1` is configured in `getSquarespaceAgentOptions.ts`
- all browser operations flow through `withNewLoggedInBrowserPage.ts` which uses the bottleneck
- even if a developer adds a new scraper, it must use the bottleneck-wrapped pattern to get an authenticated page
- the architecture makes sequential execution the path of least resistance

**shared mutable state (avoided):**
- context passed via dependency injection in `(input, context)` pattern per repo conventions
- browser cache is keyed by account + credentials (isolated per provider instance)
- grep for `let` at module level found zero matches in domain.operations or access layer
- the input-context pattern makes state isolation the default

**barrel exports (avoided):**
- grep for `export *` found zero matches in src/
- all imports use direct file paths
- rule.forbid.barrel-exports is enforced at code review level
- the repo convention makes this the established pattern

**untyped bag refs (avoided):**
- all domain refs use `RefByUnique<typeof Entity>` or `RefByUnique.as<>()`
- grep for `domain: {` without `RefByUnique` found zero matches in domain.operations
- rule.forbid.dobj.bagrefs is enforced at code review level
- domain-objects library provides the pit-of-success via typed functions

### dependencies — why they hold

**puppeteer vs playwright (avoided puppeteer):**
- package.json shows playwright-extra, not puppeteer
- stealthChromium.ts imports from playwright-extra
- playwright API is used throughout all scrapers
- no puppeteer imports exist in codebase

**mocks in tests (avoided):**
- grep for `jest.mock` found 1 match: `answerCaptchaChallenge.test.ts:111`
- that mock is for `waitForHumanSignal` — human interaction (stdin ENTER)
- human interaction cannot be tested automatically
- this exception is documented and acceptable
- all business logic tests use real dependencies or dependency injection

**direct filesystem in operations (avoided):**
- cache abstraction via `SimpleOnDiskCache` from external package
- domain.operations do not import `fs` directly
- cache logic is encapsulated in infra layer
- operations remain unit-testable without filesystem

### tech debt — why 1 item is known debt

**magic timeouts (known debt):**
- 4 instances found with fixed timeouts (2000ms, 3000ms, 1000ms, 2000ms)
- `requestTransferCode.ts:92` has `.todo` comment acknowledging fragility
- these work for current squarespace behavior but could fail on slow networks
- documented as known limitation in blueprint
- integration tests catch failures; explicit waits are future improvement

**hardcoded selectors (mitigated):**
- selectors use fallback patterns: `[data-test="x"], button:has-text("y")`
- primary selectors use data-test attributes (squarespace internal testing)
- fallbacks use text-based selectors for resilience
- integration tests catch selector drift
- centralized in selectors/*.ts files for easy updates

**implicit nullable attributes (avoided):**
- domain objects use explicit `| null` for nullable fields
- `DeclaredSquarespaceDomainRegistration.lockReason: ... | null` verified
- rule.forbid.undefined-attributes is enforced at code review level
- domain-objects library validates at construction time

### anti-patterns — why they hold

**error swallow (avoided):**
- `withNewLoggedInBrowserPage.ts` catch blocks rethrow after cleanup
- `checkSessionHealth.ts` returns failure status, does not hide errors
- no silent catch blocks that swallow exceptions
- useful-errors library used for rich error context

**create/insert verbs (avoided):**
- grep for function names with `create*` or `insert*` in domain.operations: zero matches
- all mutations use `findsert` or `upsert` semantics
- rule.forbid.nonidempotent-mutations is enforced
- the naming convention makes idempotency explicit

**positional arguments (avoided):**
- all operations use `(input, context)` pattern
- no functions with more than 2 positional args
- rule.require.input-context-pattern is enforced
- the repo convention makes this the established pattern
