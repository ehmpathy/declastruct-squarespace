### requirement 1: enumerate domains from squarespace

**why is this needed?**
because we must know which domains exist in the account before we transfer them.

**why must we know?**
because we cannot declare desired state without current state.

**why does declastruct need current state?**
because it compares declared state vs actual state to generate a plan.

**why does it compare states?**
because idempotent operations require awareness of what already exists.

**why idempotent?**
because the goal is safe, repeatable automation — not blind mutation.

**root need**: establish ground truth before mutations to enable safe declarative control.

---

### requirement 2: see dns records of domains

**why is this needed?**
because dns records might need preservation or recreation at the new registrar.

**why preserve?**
because domain transfer does not automatically transfer dns zone data.

**why not automatic?**
because dns and domain registration are separate concerns.

**why does that matter?**
because users might have custom dns records (MX, CNAME, TXT) that break if lost.

**why do they break?**
because email, website routes, and verification records are critical infrastructure.

**root need**: prevent service disruption by capture of dns configuration before transfer.

---

### requirement 3: unlock domains

**why is this needed?**
because locked domains cannot transfer to another registrar.

**why can't they?**
because transfer lock is an icann-mandated anti-hijack protection.

**why does icann mandate it?**
because domain theft via social attacks was historically common.

**why common?**
because domains are valuable assets worth theft.

**why does unlock help?**
because unlock is intentional friction — proof of legitimate owner intent.

**root need**: prove legitimate ownership intent via unlock, as required by transfer protocol.

---

### requirement 4: disable dnssec before transfer

**why is this needed?**
because dnssec-signed domains require key coordination in transfer.

**why coordination?**
because the ds record at parent zone must match keys at new registrar.

**why must they match?**
because mismatched dnssec causes complete dns resolution failure.

**why complete failure?**
because dnssec validation rejects responses signed with wrong keys.

**why is disable the solution?**
because clean disable avoids key mismatch, then re-enable after.

**root need**: avoid dns outage via clean dnssec disable before transfer.

---

### requirement 5: request transfer authorization codes

**why is this needed?**
because the target registrar requires an auth code to accept the transfer.

**why require it?**
because the auth code proves the domain owner authorized the transfer.

**why must owner prove?**
because icann requires this to prevent unauthorized transfers.

**why icann?**
because domain hijack via registrar-hop was a real attack vector.

**why does auth code work?**
because the auth code is a one-time password that proves consent.

**root need**: generate cryptographic proof of owner consent for the target registrar.

---

### requirement 6: batch operations for 300+ domains

**why is this needed?**
because manual work for 300 domains takes 50+ hours.

**why 50+ hours?**
because each domain requires ~10 minutes of clicks through squarespace ui.

**why clicks instead of api?**
because squarespace provides no api and forces manual interaction.

**why no api?**
because squarespace has no incentive to make departure easy.

**why no incentive?**
because they profit from lock-in and friction.

**root need**: overcome deliberate friction designed to prevent bulk exodus.

---

### requirement 7: playwright automation (not http api)

**why playwright?**
because squarespace provides no domain management api.

**why no api?**
because they acquired google domains and inherited a ui-only interface.

**why not build an api?**
because a public api costs resources to build and maintain.

**why accept ui-only?**
because squarespace's core business is website build, not domain services.

**why offer domains then?**
because domain services are a lock-in strategy, not a profit center.

**root need**: automate the ui since it's the only interface available.

---

### requirement 8: declarative (terraform-style) interface

**why declarative?**
because declarative is easier to reason about than imperative commands.

**why easier?**
because "desired state" is clearer than "sequence of mutations."

**why clearer?**
because compare of desired vs actual makes plans visible.

**why visible plans?**
because visible plans allow human review before execution.

**why review before execute?**
because domain operations are high-stakes — mistakes are expensive.

**root need**: reduce risk via plan-review-apply workflow for high-stakes operations.

## what i found: gaps between root needs and vision

i re-read the vision with fresh eyes, checked each root need against what's documented.

### gap 1: dns preservation is read-only

**root need**: prevent service disruption by capture of dns configuration.

**vision says**: `get.all({ domain })` scrapes dns settings page.

**gap**: the vision provides read, but the root need is preservation. read of dns is necessary but not sufficient. the user gets a snapshot but no guidance on what to do with it.

**resolution**: this is acceptable scope. the tool's job is to prepare domains for transfer from squarespace. dns import at the target registrar is out of scope. however, the vision's "usecases" table could note: "check dns records (snapshot before transfer)" to make intent clearer.

**action taken**: none. the vision correctly scopes to squarespace operations. a note would be nice but not required.

### gap 2: resumability relies on implicit idempotency

**root need**: safe automation that doesn't leave domains in bad states.

**vision says**: "idempotent operations allow safe re-run; no rollback needed"

**gap**: the vision relies on findsert/upsert semantics for resumability but doesn't explicitly call out how a partial batch is resumed. a user who hits an error at domain 150 might not realize they can just re-run.

**resolution**: the anti-patterns table already says "batch halts mid-run and can't resume → checkpoint progress; enable resumable batches". this addresses it. no gap.

### gap 3: session timeout addressed in anti-patterns

**root need**: safe automation over multi-hour batch.

**vision says**: in anti-patterns: "session expires mid-batch → health check before each operation; auto-re-login on expiry"

**found**: this was already addressed in the premortem updates. no gap.

### gap 4: hidden requirement surfaced — rate limit patterns

drill into "batch 300+ domains" revealed a root need: overcome deliberate friction.

check the anti-patterns table: "rate limit detection via patterns → randomized delays between operations; vary intervals"

**found**: addressed. the premortem identified this risk and added mitigation.
