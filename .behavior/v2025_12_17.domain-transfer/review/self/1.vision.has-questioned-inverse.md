# self-review: has-questioned-inverse

## where don't we want to end up?

the vision already includes "anti-patterns to avoid" section. review of each:

### inverse 1: domains stuck in limbo

**outcome:** domains unlocked but transfer never completed.
**how we get there:** unlock all domains, then fail before transfer code request.
**mitigation in vision:** transfer request findsert ensures code is requested after unlock.
**verdict:** ✅ addressed. the declastruct workflow handles both steps together.

### inverse 2: accidental mass-unlock of critical domains

**outcome:** production domains unlocked without intent.
**how we get there:** user runs apply without plan review.
**mitigation in vision:** plan review before apply; user declares which domains explicitly.
**verdict:** ✅ addressed. plan/apply separation is core to design.

### inverse 3: account lockout from bot detection

**outcome:** squarespace blocks automation, all 300 domains unreachable.
**how we get there:** rapid automated requests trigger security.
**mitigation in vision:** sequential with delays.
**verdict:** ⚠️ partially addressed. delays help, but stealth plugin still TODO.

### inverse 4: transfer codes expire unused

**outcome:** codes expire before user retrieves them from email.
**how we get there:** user requests codes then forgets to check email.
**mitigation in vision:** document expiration timeline.
**verdict:** ⚠️ weak mitigation. only documentation, no automation.
**note:** acceptable - email retrieval is out of scope.

### inverse 5: dns outages in transfer

**outcome:** domain resolves incorrectly after transfer.
**how we get there:** dns records not preserved, or propagation issues.
**mitigation in vision:** out of scope - registrar-to-registrar concern.
**verdict:** ✅ explicitly out of scope. dns records get operation exists for audit.

### inverse 6: partial batch failure with inconsistent state

**outcome:** 150 domains transferred, 150 not, no way to know which.
**how we get there:** process crashes mid-batch.
**mitigation in vision:** idempotent operations allow safe re-run.
**verdict:** ✅ addressed. findsert/upsert semantics enable recovery.

### inverse 7: no way to filter/exclude specific domains

**outcome:** user forced to transfer all or none.
**how we get there:** tool provides no filter mechanism.
**mitigation in vision:** user controls resources.ts declarations; can filter in code.
**verdict:** ✅ addressed. code is the filter.

### inverse 8: multiple code requests invalidate prior codes

**outcome:** new code request cancels old one, user loses track.
**how we get there:** re-run apply multiple times.
**mitigation in vision:** findsert semantics prevent duplicate requests.
**verdict:** ✅ addressed. idempotency prevents this.

## what would make this VISION fail?

not individual operations - the entire approach.

### vision failure 1: squarespace adds captcha everywhere

**outcome:** automation becomes impossible.
**likelihood:** moderate. squarespace is aware of scrapers.
**mitigation:** none in current vision. would require captcha-solving service or manual intervention.
**verdict:** ⚠️ existential risk. if this happens, vision is blocked. no fallback.
**what we accept:** this is the risk of automating a hostile platform. we document it and proceed.

### vision failure 2: squarespace removes web ui entirely

**outcome:** no surface to automate.
**likelihood:** very low. 300k+ squarespace customers use this UI.
**mitigation:** none needed - too unlikely.
**verdict:** ✅ acceptable risk.

### vision failure 3: declarative model doesn't fit

**outcome:** users confused by plan/apply for what feels like "just transfer my domains"
**likelihood:** low. terraform model is well understood.
**mitigation:** vision documents simple workflow. cli makes it easy.
**verdict:** ✅ addressed in vision ux section.

### vision failure 4: transfer codes change to in-app only

**outcome:** email flow breaks, we can't automate code retrieval.
**likelihood:** low but possible (industry trend toward app-based auth).
**mitigation:** out of scope in vision - acceptable.
**verdict:** ✅ acceptable. would require new feature (app scraping) but doesn't block current approach.

### vision failure 5: squarespace merges with different registrar

**outcome:** ui completely changes, all selectors break.
**likelihood:** possible. squarespace acquired google domains in 2023.
**mitigation:** selector maintenance documented as ongoing concern.
**verdict:** ✅ addressed. vision acknowledges ui fragility.

## what else could fail badly?

beyond vision-level failures - individual operation failures:

### inverse 9: selector failure on first domain

**outcome:** process fails immediately, user has no data.
**how we get there:** squarespace UI changed, selectors don't match.
**mitigation in vision:** mentioned in "must validate" but no graceful degradation.
**verdict:** ⚠️ gap. should fail fast with clear error, not silently scrape empty data.

### inverse 10: credentials leaked in logs

**outcome:** password/totpSecret exposed in error output.
**how we get there:** error log includes context with credentials.
**mitigation in vision:** not mentioned.
**verdict:** ⚠️ gap found and fixed. credentials should be redacted from all logs.
**action:** added to vision anti-patterns table.

### inverse 11: browser resource exhaustion

**outcome:** playwright consumes all memory in 300-domain batch.
**how we get there:** browser contexts not cleaned up.
**mitigation in vision:** not mentioned.
**verdict:** ⚠️ gap found and fixed. should reuse browser context, clean up pages.
**action:** added to vision anti-patterns table.

## findings

### vision-level risks identified

one existential risk: **captcha everywhere**. if squarespace adds captcha to all domain operations, the vision is blocked with no fallback.

**decision:** we accept this risk. the alternative (manual 50 hours) is worse. we proceed knowing automation may become impossible if squarespace actively fights it.

this is documented in vision section "open questions & assumptions" but should be more prominent. the vision acknowledges UI fragility; captcha is the ultimate expression of that fragility.

**no vision change needed.** the risk is inherent to automating a hostile platform.

### operation-level gaps identified

1. **credential redaction** - not mentioned in vision. passwords and totp secrets must never appear in logs.

2. **resource management** - 300 domains is a lot. browser resource cleanup not specified.

3. **graceful failure on selector mismatch** - should fail loudly with actionable error, not silently return empty data.

### action taken

**vision updated.** added 3 anti-patterns to 1.vision.md:

1. `credentials leaked in error logs` → redact password and totpSecret from all log output
2. `browser resource exhaustion in batch` → reuse browser context; clean up pages after each domain
3. `silent failure on selector mismatch` → fail fast with clear error; do not return empty data silently

these were gaps in the original anti-patterns table. now documented at vision level so criteria/blueprint will address them.

### why the vision now holds

the vision originally had 10 anti-patterns. my review found 3 gaps:
1. credential redaction
2. resource management
3. fail-fast on selector mismatch

**i updated the vision** to add all 3 to the anti-patterns table. vision now has 13 anti-patterns.

the vision is complete for vision-level concerns.

## conclusion

two levels of inverse analysis:

1. **vision-level:** one existential risk (captcha everywhere) accepted. no mitigation possible for a hostile platform. we proceed knowing the risk.

2. **operation-level:** found 3 gaps in anti-patterns. fixed by updating 1.vision.md with new anti-pattern rows.

vision now holds with:
- explicit acceptance of existential platform risk
- comprehensive failure mode coverage (13 anti-patterns)
- clear understanding of what we cannot control
