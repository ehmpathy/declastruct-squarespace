# self-review: has-questioned-fundamentals

## what are the fundamental truths here?

1. **squarespace has no api** — verified. they provide zero programmatic access to domain management. this is the root constraint.

2. **300+ domains need to transfer** — this is the actual problem. without scale, manual work is acceptable.

3. **transfer requires specific steps** — unlock domain, disable dnssec, request auth code. these are icann/registrar requirements, not squarespace-specific.

4. **browser automation can interact with web UIs** — playwright/puppeteer exist and work. this is established technology.

5. **squarespace uses standard web patterns** — login, session cookies, html forms. no exotic mechanisms.

## do we build from fundamentals or convention?

**convention used:**
- terraform mental model (plan/apply) — familiar but not fundamental
- declastruct pattern (declarative daos) — our convention, adds complexity
- domain-objects library — our pattern for type safety

**from fundamentals:**
- "automate browser clicks because no api exists" — this is first-principles
- "cache remote state because scraping is slow" — first-principles optimization
- "idempotent operations because retries happen" — first-principles reliability

**verdict:** the core insight (playwright automation) is fundamental. the declastruct wrapper is convention for consistency with our other tools.

## if we started from scratch?

minimum viable:
```sh
# what if we just...
node transfer-all.js --email x --password y --totp-secret z
```

single command that:
1. logs in
2. iterates all domains
3. unlocks each
4. disables dnssec
5. requests transfer code
6. logs progress

**would this be simpler?** yes, initially.

**would this be better?** no, because:
- no plan/preview step — risky for 300 domains
- no idempotency — fails mid-batch require manual tracking
- no caching — every retry re-scrapes everything
- no reusability — one-off procedure vs library

**the declastruct layer is justified** for this scale and risk level.

## cost breakdown (raw materials)

| component | what it provides | could we skip it? |
|-----------|------------------|-------------------|
| playwright | browser automation | no — core requirement |
| totp generation | 2fa handling | no — squarespace requires it |
| scraper functions | extract data from pages | no — needed to read state |
| domain objects | type safety, refs | optional — adds value but not required |
| dao pattern | get/set abstraction | optional — adds consistency |
| cache layer | performance | optional — but needed for 300 domains |
| declastruct provider | plan/apply integration | optional — but enables familiar workflow |

**fundamental cost:** playwright + scrapers + totp
**optional layers:** domain-objects, daos, caching, declastruct

we chose the full stack for consistency with declastruct-aws patterns. this is a tradeoff: more upfront complexity for familiar interfaces and reusability.

## conclusion

the vision builds from both fundamentals and conventions:
- fundamental: browser automation because no api
- convention: declastruct pattern for declarative control

the convention is justified by scale (300 domains) and risk (don't want to accidentally unlock everything without review).

if this were a 5-domain problem, a simple command would suffice. at 300 domains, the plan/apply workflow earns its complexity.
