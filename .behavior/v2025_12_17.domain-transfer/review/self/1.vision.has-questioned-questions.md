## self-answered analysis

### 2fa totp reliability

**question**: does totp integration work reliably?

**analysis**:
- totp is a standard algorithm (RFC 6238)
- playwright can wait for 2fa challenge, enter code, submit
- the totpSecret allows code generation without human intervention
- failure modes: clock drift (30-sec window is tolerant), incorrect secret
- mitigation: fail fast if totp rejected, prompt user to verify secret

**answer**: totp is well-understood. use `otplib` or similar library. if totp fails, fail fast with clear error. no vision change needed.

### cache TTL appropriateness

**question**: is cache TTL appropriate for 300+ domain accounts?

**analysis**:
- domain registrations rarely change intra-day
- use case is batch migration: read state → plan → apply
- stale cache worst case: plan shows outdated state, user re-runs
- cache invalidation on mutation ensures accuracy after changes

**answer**: 24-hour default TTL is reasonable. add cache-clear option in config for edge cases. no vision change needed — vision already says "cache TTL, can be cleared via config".

### registrar types and UI consistency

**question**: do different registrar types (SQUARESPACE_DOMAINS_LLC, TUCOWS, KEY_SYSTEMS) have the same transfer UI?

**analysis via web research**:
- squarespace acquired google domains in 2023
- google domains used multiple backend registrars
- squarespace help docs mention different verification email sources per registrar
- core transfer UI (unlock → request code) likely same
- edge behaviors may differ: lock periods, verification flows

**answer**: core transfer flow likely consistent. edge behaviors may vary. the vision already surfaces `registrarType` in domain entity. this is appropriate for validation in implementation.
