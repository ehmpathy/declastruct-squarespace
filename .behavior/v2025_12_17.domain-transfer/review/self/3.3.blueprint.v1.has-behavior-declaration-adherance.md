## deviations found

### deviation 1: dns field name

**spec:** criteria says "each record includes type, name, value, ttl"
**implementation:** uses `host` instead of `name`

**analysis:** this is correct DNS terminology. the spec used "name" loosely; `host` is the standard term for the subdomain portion.

**verdict:** acceptable deviation (follows domain convention)

### deviation 2: browser endpoint property name

**spec:** vision line 104 says `browser?: { wsEndpointExtant?: string }`
**implementation:** `getDeclastructSquarespaceProvider.ts` line 58 says `extantBrowserWSEndpoint`

**analysis:** both names describe the same concept (a pre-launched browser websocket endpoint). the implementation uses a more explicit name.

**verdict:** acceptable deviation (clearer name, same function)

### deviation 3: dns dao uses string instead of RefByUnique

**spec:** vision line 75 says `daos.DeclaredSquarespaceDomainDnsRecord.get.all({ domain: { name } })`
**implementation:** `DeclaredSquarespaceDomainDnsRecordDao.get.all({ domainName: string })`

**analysis:** spec uses object shape `{ domain: { name } }` which implies RefByUnique pattern. implementation uses plain string `domainName`.

**verdict:** deviation from spec - should use `domain: RefByUnique<typeof DeclaredSquarespaceDomainRegistration>`

### deviation 4: lock validation on transfer

**spec:** criteria says locked domain should return "error: domain must be unlocked first"
**implementation:** no explicit lock check; relies on Squarespace UI

**analysis:** documented in coverage review. Squarespace blocks the action at UI level.

**verdict:** minor gap - explicit check recommended for better UX
