# self-review: has-questioned-premortem

## stone: 3.3.blueprint.v1

---

## the scenario

it's 3 months after deployment. the batch transfer of 300 domains failed catastrophically. what went wrong?

---

## failure mode 1: bot detection at domain 147

**what happened:**
squarespace detected automated access after 147 domains processed. account was locked for 24 hours. user had to call support, explain they weren't a bot, and start over.

**why it failed:**
- playwright-stealth was NOT installed (documented in has-questioned-assumptions)
- only mitigation was custom userAgent — insufficient against modern bot detection
- cloudflare or similar service flagged headless browser fingerprint

**technical assumption that was wrong:**
custom userAgent + 500ms delay = sufficient evasion

**how to address:**
1. add playwright-stealth to dependencies
2. implement stealth context setup in genLoggedInBrowser.ts
3. add random jitter to 500ms delay (400-800ms range)

**addressed?** NO — requires code changes

---

## failure mode 2: selector breakage after ui update

**what happened:**
squarespace shipped a ui redesign. all scrapers returned empty results. domains showed as no dns records. plan showed "delete all records" for every domain.

**why it failed:**
- selectors assumed specific data-test attributes
- fallback class selectors also changed
- no alert when scrape returns unexpected empty results

**technical assumption that was wrong:**
data-test attributes are stable indicators

**how to address:**
1. add smoke test that runs daily to detect selector drift
2. add sanity check: if domain.list returns 0 but account has domains, throw error
3. document known selector brittleness in readme

**addressed?** PARTIALLY — error thrown when domain not found, but empty list passes silently

---

## failure mode 3: session expired at hour 2.5 of batch

**what happened:**
browser cache expired after 1 hour. session was still valid but cache thought it was stale. new browser launched mid-batch. login failed due to ip change that triggered 2fa. batch crashed.

**why it failed:**
- browser cache expiration (1 hour) vs batch duration (2-3 hours) mismatch
- no session refresh mechanism
- handleReauthentication only handles modal, not full login redirect

**technical assumption that was wrong:**
browser cache expiration equals session expiration

**how to address:**
1. extend browser cache to 4 hours for batch operations
2. add proactive session check before each scrape
3. consider: disable browser cache expiration while batch is active

**addressed?** YES — browser cache extended to 4 hours in getSquarespaceAgentOptions.ts

---

## failure mode 4: tucows domain has different transfer ui

**what happened:**
first 200 domains (squarespace registrar) transferred fine. next 100 domains (tucows backend) had different ui flow — transfer button in different location. requestTransferCode returned success: false. user didn't notice until emails didn't arrive.

**why it failed:**
- integration tests only covered squarespace-registered domains
- no test coverage for tucows or key_systems registrar types
- registrar type is parsed but ui differences not validated

**technical assumption that was wrong:**
all registrar backends use identical squarespace ui

**how to address:**
1. add integration tests with domains from each registrar type
2. add warn in logs when registrar is not SQUARESPACE_DOMAINS_LLC
3. document potential ui differences in readme

**addressed?** NO — requires test account with multiple registrar domains

---

## failure mode 5: 60-day lock not detected, unlock silently failed

**what happened:**
20 domains were within 60-day icann lock period. toggleDomainLock was called, returned without error, but domain remained locked. transfer code request proceeded but squarespace rejected it. user only discovered when transfers failed at destination registrar.

**why it failed:**
- lockReason field not checked before unlock attempt
- toggleDomainLock assumed click success = state change
- no verification read after mutation

**technical assumption that was wrong:**
successful ui interaction = successful state change

**how to address:**
1. check lockReason before attempt unlock — if 60-day lock, throw BadRequestError
2. add verification: re-scrape domain after mutation, confirm state changed
3. document 60-day lock in vision (already done)

**addressed?** YES — lockReason pre-check added to setDomain.ts; throws BadRequestError with hint

---

## failure mode 6: stale cache caused incorrect plan

**what happened:**
user ran `declastruct plan` at 8am. cache populated with unlocked domains. at noon, squarespace auto-locked domains for security event. user ran `declastruct apply` at 2pm. plan used stale cache. apply tried to unlock already-unlocked domains (per cache). mutations failed silently. user thought apply succeeded.

**why it failed:**
- 24-hour cache ttl too long for active use
- no cache invalidation hint in cli
- mutations don't verify pre-state matches expected

**technical assumption that was wrong:**
domain state is stable for 24 hours

**how to address:**
1. add `--no-cache` flag to bypass cache
2. add cache age warn: "data is 6 hours old, consider refresh"
3. add pre-mutation state verification

**addressed?** NO — requires cli enhancements

---

## failure mode 7: email delivery silent failure

**what happened:**
transfer codes were requested successfully (requestTransferCode returned success: true). emails never arrived — spam filter, typo in admin email, squarespace email service outage. user waited 3 days. transfer codes expired. had to re-request.

**why it failed:**
- email delivery is out of scope (external dependency)
- no way to verify email was received
- 7-day auth code expiration not documented

**technical assumption that was wrong:**
squarespace email delivery is reliable

**how to address:**
1. document email delivery risk in readme
2. add cli hint: "check email within 24 hours"
3. consider: log expected email recipient for user verification

**addressed?** PARTIALLY — documented in vision that auth code goes to email
