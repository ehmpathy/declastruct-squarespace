### usecase.1 = enumerate domains

**devil's advocate**: "why enumerate all domains? just let users specify which domains they want to transfer."

**response**: users don't always know their full inventory. with 300+ domains across years of registration, enumeration is essential for discovery. a user who forgot about domain X shouldn't lose it.

**too strict/loose?** appropriate. returns all domains, lets user filter in code.

**verdict**: criterion stands.

---

### usecase.2 = check dns records

**devil's advocate**: "dns records are irrelevant — the new registrar will have fresh dns. why preserve old records?"

**response**: users often have MX records for email, CNAME for services, TXT for verification. loss of these causes email failure, site outage, or verification loss. dns snapshot is critical pre-transfer.

**too strict/loose?** appropriate. captures all record types.

**verdict**: criterion stands.

---

### usecase.3 = unlock domain for transfer

**devil's advocate**: "why test the 60-day lock? it's an edge case that affects few domains."

**response**: the 60-day lock is icann-mandated for recent transfers or contact changes. if ignored, this leads to silent failures on those domains. the criterion ensures users get clear feedback.

**too strict/loose?** appropriate. tests the main case and the icann restriction case.

**verdict**: criterion stands.

---

### usecase.4 = disable dnssec

**devil's advocate**: "dnssec is rare. most domains don't use it. why have a dedicated criterion?"

**response**: dnssec adoption is ~15% for .com domains. for 300 domains, that's ~45 domains. dnssec-enabled domains that transfer without disable cause total dns failure. the criterion prevents this.

**too strict/loose?** appropriate. the idempotent case handles domains without dnssec.

**verdict**: criterion stands.

---

### usecase.5 = request transfer authorization code

**devil's advocate**: "the transfer code is emailed anyway. why track the request as an entity?"

**response**: idempotency. if a user runs the batch twice (intentionally or due to failure), the system should NOT send duplicate emails. the entity tracks "already requested" state.

**criticism accepted**: the criterion could be clearer about what "returns transfer request entity" means. does it include a timestamp? a status?

**action considered**: add clarity to criterion about what the entity contains.

**verdict**: criterion stands, but could be more specific.

---

### usecase.6 = batch domain transfer workflow

**devil's advocate**: "300 domains is arbitrary. why not test with 5? or 1000?"

**response**: 300 is from the wish. the number is illustrative. the criterion tests batch behavior, not a specific count.

**second criticism**: "plan shows required changes for each domain" — what if 150 domains need changes and 150 don't? is that testable?

**response**: the criterion covers "completed domains are skipped (idempotent)" in the failure recovery scenario. this implicitly tests the "no change needed" case.

**verdict**: criterion stands.

---

### usecase.7 = authentication

**devil's advocate**: "why test sms and passkey rejection? just document that only totp is supported."

**response**: documentation is not enforcement. if a user with sms 2fa tries to use the tool, they should get a clear error, not a cryptic failure in the 2fa challenge.

**second criticism**: "session expires mid-batch → session is automatically refreshed" — is this testable in blackbox?

**response**: yes. the blackbox test would run a long operation and verify it completes. if session expired and wasn't refreshed, the operation would fail.

**verdict**: criterion stands.

---

### usecase.8 = bot detection resilience

**devil's advocate**: "you cannot guarantee 'operations complete without bot detection blocks' — squarespace can add captcha anytime."

**response**: correct. this criterion tests current behavior, not future-proofed behavior. if squarespace adds captcha, the test fails and we know to address it. that's the point — the test is a canary.

**second criticism**: "execution pace avoids rate-based detection" is not testable in blackbox.

**response**: it IS testable in blackbox. run 10 operations sequentially. if they all succeed, the pace was acceptable. the HOW (delays, randomization) is blueprint; the WHAT (operations succeed) is blackbox.

**verdict**: criterion stands.

**criticism**: every criterion assumes success. what about partial success? what if 299/300 domains transfer but 1 fails?

**response**: usecase.6 covers "batch fails mid-execution" and "partial failures are recoverable". the failure case is tested.

**criticism**: "1-3 hours" for 300 domains is in the vision. the criteria don't verify this.

**response**: duration is performance, not behavior. blackbox tests the outcome, not the duration. if duration matters, add a performance criterion separately.

**verdict**: this is a valid criticism. however, performance criteria are typically separate from functional criteria. no change needed for this blackbox document.

### critic 3: "these criteria don't test observability"

**criticism**: how does the user know progress? the criterion says "progress is visible" but doesn't specify what that means.

**response**: "progress is visible" is intentionally abstract at blackbox level. the blueprint will specify the mechanism (logs, progress bar, etc.). blackbox tests the outcome — user can observe progress — not the mechanism.

**verdict**: appropriate for blackbox. no change needed.

## harder challenge: exposing weak criteria

i paused and re-read the criteria with hostile intent. which criterion would fail first under real-world conditions?

### weak spot identified: usecase.6 recovery scenario

the criterion says:

```
given('batch fails mid-execution at domain 150')
  when('user re-applies the plan')
    then('completed domains are skipped (idempotent)')
    then('failed domain is retried')
    then('rest of domains are processed')
```

**devil's advocate**: "what if the failure was at domain 150 but domains 151-200 were ALSO in a bad state because of a squarespace outage? the criterion assumes only one domain fails."

**analysis**: this is a valid concern. a squarespace outage could fail multiple domains simultaneously. the criterion tests single-domain retry, not cascading failure recovery.

**resolution**: the idempotent semantics handle this. if domains 150-200 all failed, re-run would retry ALL of them because none are in "completed" state. the criterion "completed domains are skipped" implies non-completed domains are processed.

**verdict**: criterion is sufficient. cascading failure is handled by the same idempotent logic.

### weak spot identified: usecase.8 testability

the criterion says:

```
given('squarespace may detect automated access')
  when('operations are executed')
    then('operations complete without bot detection blocks')
```

**devil's advocate**: "how do you test that bot detection was NOT triggered vs bot detection didn't exist? the test can't distinguish 'we passed stealth checks' from 'there were no stealth checks to pass'."

**analysis**: this is philosophically true. we cannot prove a negative. however, the criterion is still valuable:
- if squarespace adds bot detection, this test will fail
- the test serves as a regression canary
- "passes today" is sufficient for "works today"

**verdict**: criterion is appropriate for blackbox. we accept the epistemological limitation.

## fresh review 2026-03-12

re-read all 8 usecases line by line. paused on each criterion to ask: why might this be wrong?

---

### usecase.1 — cache as blackbox concern?

**devil's argument**: cache is implementation detail. users don't care about cache — they care about results.

**why it holds**: the criterion says "result is cached for subsequent calls sothat repeated queries are fast". this is observable behavior at blackbox level: call get.all() twice, second call is faster. users DO care about this — a batch tool that re-scrapes 300 domains on every call would be unusable.

cache behavior is user-facing contract, not hidden implementation.

---

### usecase.2 — empty/default ambiguity

**devil's argument**: "empty list or default records only" — which is it? tests need specificity.

**why it holds**: squarespace may inject preset records (squarespace hosting A records, for example) that the user never created. the criterion captures the essential behavior: "user knows no custom records need preservation". whether that manifests as empty list or default-only list is squarespace's choice, not ours.

the ambiguity reflects reality: we don't control squarespace's defaults.

---

### usecase.3 — 60-day lock common enough?

**devil's argument**: most users have stable domains. 60-day lock is edge case.

**why it holds**: any user who:
- recently registered a domain
- recently transferred a domain in
- recently changed WHOIS contact info

will hit the 60-day lock. for a batch transfer tool, this is NOT edge case — it's the primary pain point. many users discover they CAN'T transfer domains they assumed were transferable.

the criterion ensures clear feedback when this happens.

---

### usecase.4 — dnssec adoption too low?

**devil's argument**: ~15% adoption means 85% of domains don't need this criterion.

**why it holds**: for 300 domains, 15% = ~45 domains. if any of those transfer with dnssec enabled and the new registrar doesn't have the DS record, dns resolution fails completely. not "works partially" — fails entirely.

the cost of criterion: one extra check before transfer.
the cost of omission: potential complete dns failure for ~45 domains.

risk/benefit is clear.

---

### usecase.5 — 14-day expiration accuracy

**devil's argument**: if actual expiration is 7 days or 30 days, the criterion is wrong.

**why it holds**: the exact number doesn't matter for blackbox criteria. what matters is the behavior: "expired codes need refresh, fresh codes should not re-request". the criterion tests the behavior pattern, not the specific TTL.

if squarespace uses 7 days or 30 days, the implementation adapts; the criterion describes what should happen.

---

### usecase.6 — "visible" undefined

**devil's argument**: "progress is visible" is not testable. visible how? to whom?

**why it holds**: blackbox tests observable behavior. "visible" means the user can observe the batch is in progress. this could be:
- log output
- progress bar
- status API
- file written

the HOW is blueprint concern. the WHAT (user can observe progress) is blackbox concern. test: during batch execution, can user determine batch is active? yes/no is testable.

---

### usecase.7 — account lock scenario

**devil's argument**: squarespace might lock account for suspicious activity. should test this.

**why it holds**: squarespace presents account lock the same as wrong password — both result in login failure. the criterion "returns error: authentication failed" covers both cases. testing account lock specifically would require squarespace cooperation (triggering a lock), which is not practical.

the criterion tests what we can observe: login succeeds or fails.

---

### usecase.8a — stealth testability

**devil's argument**: "masks automation signals" — how do you test absence of detection?

**why it holds**: observable via `navigator.webdriver` property:
- without stealth: `navigator.webdriver === true`
- with stealth: `navigator.webdriver === false`

this is not testing absence — it's testing presence of the mask. the mask is observable. whether squarespace checks the mask is their concern; the criterion tests we applied it.

---

### usecase.8b — turnstile structure change

**devil's argument**: cloudflare changes turnstile; criteria become obsolete.

**why it holds**: criteria describe behavioral contracts:
1. detect challenge
2. pivot mode
3. solve (click or prompt)
4. resume

if cloudflare changes turnstile from checkbox to slider, the behavior is same: detect → pivot → solve → resume. implementation changes; criteria describe intent.

if cloudflare creates entirely new challenge type, criteria would need update. but "cloudflare might change everything" applies to all automation — we can't future-proof against complete redesign.
