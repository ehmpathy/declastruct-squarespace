## technical assumptions surfaced (with code evidence)

### assumption 1: squarespace ui selectors will remain stable

**what we assume:** css selectors will continue to work

**what if opposite true:** squarespace redesigns ui, all selectors break

**evidence from code:**
```ts
// src/access/sdks/squarespace.via.playwright/selectors/domainDetailSelectors.ts
container: '[data-test="domain-detail"], .domain-detail',
lockStatus: '[data-test="lock-status"], .lock-status',
```
selectors use `[data-test="..."], .class-fallback` pattern throughout.

**validated:** yes — fallback pattern is implemented. but data-test attributes are not guaranteed by squarespace (they could be internal test hooks removed in production).

**risk level:** medium — fallback to class selectors helps, but squarespace has no stability contract for either.

---

### assumption 2: playwright-stealth is used for bot detection evasion

**what we assume:** playwright-stealth patches automation signals (per vision line 165, 189)

**what if opposite true:** no stealth plugin, bot detection risk is higher than documented

**evidence from code:**
- `package.json`: NO playwright-stealth in dependencies
- grep for "playwright-stealth" in src/: NO matches
- `genLoggedInBrowser.ts` line 33-37: only sets custom userAgent

```ts
// actual bot mitigation (genLoggedInBrowser.ts:33-37)
const context = await browser.newContext({
  userAgent: 'Mozilla/5.0 (Macintosh; Intel Mac OS X 10_15_7)...',
  viewport: { width: 1920, height: 1080 },
});
```

**validated:** NO — vision and blueprint claim playwright-stealth is used, but it is NOT installed or imported. this is a documentation-to-implementation gap.

**risk level:** HIGH — bot detection evasion is weaker than documented. the only mitigation is a static userAgent string.

**action needed:** either add playwright-stealth or update documentation to reflect actual implementation.

### assumption 3: sequential operations prevent bot detection

**what we assume:** bottleneck rate limit (500ms delay, maxConcurrent: 1) prevents detection

**what if opposite true:** squarespace uses more sophisticated detection (fingerprint, behavior analysis)

**evidence from code:**
```ts
// getSquarespaceAgentOptions.ts:52-54
const bottleneck = new Bottleneck({
  maxConcurrent: 1,
  minTime: 500, // 500ms between operations
});
```

**validated:** yes — rate limit is implemented as documented.

**counterexamples:** cloudflare, akamai use fingerprint analysis beyond pace. squarespace may use similar. given playwright-stealth is NOT installed (assumption 2), fingerprint mitigation is absent.

**risk level:** medium-high — pace is controlled but fingerprint evasion is absent.

### assumption 6: browser session survives multi-hour batch

**what we assume:** login session persists for 300+ domain batch (1-3 hours)

**what if opposite true:** session expires mid-batch, operations fail

**evidence from code:**
```ts
// getSquarespaceAgentOptions.ts:42-44
const browserCache = createInMemoryCache<LoggedInBrowser>({
  expiration: { hours: 1 },
});
```

**gap found:** browser cache expires after 1 hour, but batch for 300 domains takes 1-3 hours. if session expires, subsequent operations would fail.

**mitigation present:**
```ts
// handleReauthentication.ts:11-36
export const handleReauthentication = async (page, credentials) => {
  const reauthModal = await page.$(authSelectors.reauthModal.container);
  if (!reauthModal) return false;
  // enters password in reauth modal...
}
```

**validated:** partially — reauth modal handle exists but is not automatically invoked on session expiry. scrapers would need to call it proactively.

**risk level:** medium — needs validation that handleReauthentication is called on scrape failure.

## conclusion

2 critical findings from code review:

1. **playwright-stealth gap**: vision and blueprint claim it's used, but it's NOT in dependencies. bot detection evasion is weaker than documented.

2. **registrar type coverage gap**: integration tests only cover one registrar type. TUCOWS and KEY_SYSTEMS domains are not tested.

1 implementation gap found:

3. **session refresh gap**: browser cache expires after 1 hour but handleReauthentication may not be called automatically on failure.

these gaps should be addressed before production use with 300+ domains.
