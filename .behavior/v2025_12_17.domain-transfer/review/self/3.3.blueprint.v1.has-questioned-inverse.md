## what architectural states must we avoid?

### avoid: partially processed batch with no recovery point

**the anti-state:**
batch processes 150/300 domains, crashes, no record of which domains completed. user re-runs, duplicates work or creates inconsistent state.

**current blueprint status:**
- idempotent operations (findsert, upsert) prevent double-apply
- no explicit checkpoint/resume mechanism
- acceptable because idempotency covers recovery

**verdict:** not a concern — idempotency handles partial recovery naturally.

### avoid: divergent cache and remote state

**the anti-state:**
cache says domain is unlocked. remote (squarespace) says domain is locked. plan shows "no changes needed" when changes are needed.

**current blueprint status:**
- 24-hour cache ttl is long
- mutations invalidate cache
- no staleness detection

**verdict:** risk documented in premortem. acceptable for one-time migration — user can clear cache.

### avoid: browser session leak

**the anti-state:**
browser processes stay open after batch, consume memory. user runs multiple times, spawns zombie chromium processes.

**current blueprint status:**
- browser.close() in finally block
- page cleanup on error
- bottleneck ensures sequential (no parallel leaks)

**verdict:** covered — cleanup paths exist.

## what dependencies would create problems?

### avoid: puppeteer alongside playwright

**why bad:**
- both control chromium
- driver version conflict
- bloated dependencies

**current blueprint:**
playwright only. no puppeteer.

**verdict:** clean.

### avoid: screen record or video libraries

**why bad:**
- increases memory usage
- larger attack surface
- unnecessary for batch transfer

**current blueprint:**
no video, no screenshots.

**verdict:** clean.

### avoid: external dns api dependencies

**why bad:**
- scope creep beyond squarespace
- transfer verification is out-of-scope
- introduces new failure modes

**current blueprint:**
dns records are read-only snapshot. no external dns mutation.

**verdict:** clean.

## how could this design accidentally create tech debt?

### tech debt risk 1: selector sprawl

**how it happens:**
each ui flow adds more selectors. selectors accumulate in 4 files. squarespace updates break random subset. maintenance burden grows.

**mitigation in blueprint:**
- selectors centralized in /selectors/ folder
- each file corresponds to one page type
- fallback pattern (data-test, class) reduces single-point failures

**verdict:** acceptable. centralization helps. alternative (visual test) is heavier.

### tech debt risk 2: hardcoded delays

**how it happens:**
`waitForTimeout(3000)` and `minTime: 500` are magic numbers. no visibility into why these values. future maintainer changes them, breaks things.

**mitigation applied:**
- added .why comments for 500ms delay in getSquarespaceAgentOptions.ts
- added .note and .todo for 3000ms timeout in requestTransferCode.ts

**verdict:** addressed — delays now documented in code.

### tech debt risk 3: registrar-agnostic code with registrar-specific behavior

**how it happens:**
code treats all registrar types identically. tucows domains have subtle ui differences. code works for 80% of domains, fails mysteriously for 20%.

**mitigation needed:**
- add registrar type to scrape results
- add integration tests per registrar type
- consider: registrar-specific selector overrides

**verdict:** documented risk. premortem identifies this as unaddressed.

## what anti-patterns should we steer clear of?

### anti-pattern: shared browser context across accounts

**why bad:**
- session bleed between accounts
- security risk
- cookie confusion

**blueprint avoids this:**
browser cache keyed by `[account.id, credentials.email]`. separate browsers per account.

**verdict:** avoided.

### anti-pattern: parallel ui operations

**why bad:**
- bot detection
- race conditions in spa
- ui state corruption

**blueprint avoids this:**
bottleneck `maxConcurrent: 1`. all operations sequential.

**verdict:** avoided.

### anti-pattern: store or return auth codes

**why bad:**
- security liability (auth codes are secrets)
- codes expire (7 days), stale data
- squarespace sends via email (out of band)

**blueprint avoids this:**
`transferCode: null` when email sent. code not stored. user retrieves from email.

**verdict:** avoided.

### anti-pattern: retry loops with exponential backoff on ui failures

**why bad:**
- ui failures often require human intervention
- backoff doesn't help with selector breakage
- can trigger rate limits

**blueprint avoids this:**
single attempt, then fail. no retry loop in scrapers. idempotency enables manual re-run.

**verdict:** avoided.

### anti-pattern: mock squarespace responses in tests

**why bad:**
- mocks drift from reality
- false confidence
- hides selector breakage

**blueprint avoids this:**
integration tests use real squarespace account. no mocks.

**verdict:** avoided.
