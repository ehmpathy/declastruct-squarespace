# self-review: has-questioned-assumptions

## assumptions extracted from vision

the vision explicitly lists assumptions (section: "open questions & assumptions"). review of each:

### assumption 1: squarespace ui selectors remain stable

**what we assume:** ui selectors we encode will continue to work.
**evidence:** squarespace ui has been relatively stable. but no guarantee.
**if opposite true:** ui redesign breaks all scrapers. maintenance required.
**wisher said?** no. we inferred this risk.
**verdict:** ⚠️ true assumption. vision acknowledges fragility. acceptable risk.

### assumption 2: one account at a time

**what we assume:** single account per provider instance.
**evidence:** wish mentions "300+ domains" singular. implies one account.
**if opposite true:** need multi-account support. more complex provider.
**wisher said?** confirmed one account is fine.
**verdict:** ✅ validated. explicit confirmation.

### assumption 3: transfer codes via email is acceptable

**what we assume:** user will retrieve codes from email manually.
**evidence:** squarespace sends codes via email. no programmatic retrieval option.
**if opposite true:** need email integration (imap scrape). much more complex.
**wisher said?** confirmed acceptable.
**verdict:** ✅ validated. explicit confirmation.

### assumption 4: chromium-based automation is acceptable

**what we assume:** playwright/chromium overhead is acceptable.
**evidence:** no lightweight http-only option extant for squarespace.
**if opposite true:** need to discover and use internal ajax apis. fragile.
**wisher said?** implicit acceptance by agreement with playwright approach.
**verdict:** ✅ reasonable. no alternative.

### assumption 5: stealth plugin required

**what we assume:** need playwright-extra + stealth plugin for bot detection.
**evidence:** vision notes this is TODO. currently only userAgent set.
**if opposite true:** bot detection may not exist or may not trigger.
**wisher said?** no. we assumed based on best practice.
**verdict:** ⚠️ unvalidated. should test without stealth first. add if needed.

### assumption 6: totp 2fa only

**what we assume:** user has totp configured, not sms or passkey.
**evidence:** vision states this as constraint. automation cannot handle sms/passkey.
**if opposite true:** user must reconfigure 2fa or cannot use tool.
**wisher said?** confirmed totp is acceptable constraint.
**verdict:** ✅ validated. explicit constraint.

## findings

### concrete finding: selector assumptions unvalidated

reviewed the actual scraper code. found pattern:
```ts
domainRow: '[data-test="domain-row"], .domain-row'
```

this assumes:
1. squarespace uses `data-test` attributes, OR
2. fallback class selectors (`.domain-row`) will match

**evidence reviewed:**
- `src/access/sdks/squarespace.via.playwright/selectors/*.ts` - all use this pattern
- pattern is sensible (data-test first, fallback second)
- but no evidence squarespace UI actually has these attributes or classes

**is this documented?** yes. vision line 213: "[ ] playwright selectors work against current squarespace ui"

**action taken:** none needed. vision already documents this as "must validate". the assumption is explicit, not hidden.

### gap noted: no failure strategy specified

the vision's "must validate" section lists risks (captcha, session timeout, selectors) but doesn't specify how we handle them if validation fails.

**action taken:** none. the vision is at design phase. failure strategies belong in criteria/blueprint. this is acceptable for vision stage.

### why the vision holds

1. **most assumptions are validated** — wisher confirmed key constraints (totp, one account, email codes, sequential).

2. **unvalidated risks are documented** — captcha, session timeout, stealth all noted in "must validate" section.

3. **no hidden assumptions that change design** — all assumptions trace to either explicit wisher confirmation or documented risks.

4. **fallback paths exist** — if stealth not needed, skip it. if captcha triggers, add solver. if session expires, add refresh. none of these invalidate the approach.

## conclusion

assumptions are either validated with wisher or documented as risks to validate in implementation. no hidden assumptions found that would change the vision. vision holds.
