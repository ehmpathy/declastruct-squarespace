## hidden assumptions discovered

### assumption 1: fixed timeout for success detection

**location:** `requestTransferCode.ts` lines 90-92

**the assumption:**
```ts
// .note - fixed timeout because success elements vary by registrar
// .todo - replace with waitForSelector for known success indicators
await page.waitForTimeout(3000);
```

**what if wrong?**
- 3 seconds too short: success element renders after 5+ seconds
- 3 seconds too long: wasted time on every operation
- network conditions vary: 3 seconds fails on slow connections

**evidence against assumption:**
- the `.todo` comment acknowledges this is fragile
- no integration test verifies this timeout is sufficient
- different registrar types (SQUARESPACE_DOMAINS_LLC, TUCOWS, KEY_SYSTEMS) may have different render times

**verdict:** acknowledged technical debt. document in blueprint.

**fix applied:** added "fragile timeout" to blueprint known limitations.

### assumption 3: cast function handles all squarespace status text

**location:** `castIntoDeclaredSquarespaceDomainTransferRequest.ts` lines 12-38

**the assumption:**
```ts
const mapRawStatusToTransferStatus = (input) => {
  if (status.includes('complete') || status.includes('transferred')) return 'COMPLETED';
  if (status.includes('cancel') || status.includes('failed')) return 'CANCELLED';
  if (status.includes('progress') || status.includes('pending')) return 'IN_PROGRESS';
  if (status.includes('code') || status.includes('sent')) return 'CODE_SENT';
  return 'REQUESTED'; // default fallback
};
```

**what if wrong?**
- what if squarespace shows "rejected" text? would map to REQUESTED incorrectly
- what if squarespace shows "expired" text? would map to REQUESTED incorrectly
- default fallback to REQUESTED masks unrecognized statuses

**evidence against assumption:**
- literal file has 5 statuses: REQUESTED, CODE_SENT, IN_PROGRESS, COMPLETED, CANCELLED
- "rejected" text not handled - would silently fall through to REQUESTED
- no fail-fast for unrecognized status text

**analysis:** the upsert logic in setTransferRequest.ts is actually correct:
- if status is COMPLETED or CANCELLED → allow re-request
- if status is REQUESTED/CODE_SENT/IN_PROGRESS → return found (don't duplicate)

the issue was in the cast function, not upsert. unrecognized status text mapped to REQUESTED silently.

**verdict:** real gap in cast function. fixed with explicit status mappings.

**fix applied:** updated cast function to handle "rejected" and "expired" status text:
```ts
// handle terminal states that allow re-request (cancelled, failed, rejected, expired)
if (
  status.includes('cancel') ||
  status.includes('failed') ||
  status.includes('reject') ||
  status.includes('expir')
)
  return 'CANCELLED';
```

test cases added to verify the fix.

### assumption 5: getAll then filter is efficient

**location:** `getOneDnsRecord.ts` lines 28-39, `setTransferRequest.ts` line 37

**the assumption:**
```ts
const dnsRecords = await getAllDnsRecords({ domain }, context);
const recordFound = dnsRecords.find((record) => /* filter */);
```

**what if wrong?**
- for domains with hundreds of DNS records: inefficient
- for accounts with hundreds of transfer requests: inefficient
- cache mitigates this, but first call is slow

**evidence against assumption:**
- typical DNS record count: 10-50 per domain (acceptable)
- typical transfer request count: 0-10 per account (acceptable)
- cache layer means subsequent calls are fast

**verdict:** acceptable for wish scope (300 domains). would not scale to thousands.
