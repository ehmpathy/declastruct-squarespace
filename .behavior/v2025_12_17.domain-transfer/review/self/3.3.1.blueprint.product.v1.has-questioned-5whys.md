## decision 1: playwright for browser automation

**why playwright?**
1. why this approach? → because squarespace has no public api
2. why? → because squarespace is hostile to automation and developer tools
3. why? → because their business model depends on lock-in
4. why? → because domain migrations generate support costs and churn
5. why? → because they profit from inertia, not from easy transfers

**root reason:** squarespace intentionally lacks an api. browser automation is the only option.

**verdict:** correct choice. no alternative exists.

---

## decision 2: sequential operations (not parallel)

**why sequential?**
1. why this approach? → because parallel requests risk rate limits and bot detection
2. why? → because squarespace monitors for automated access patterns
3. why? → because abuse protection is standard for web platforms
4. why? → because parallel requests from one IP create obvious patterns
5. why? → because we need the batch to complete reliably over speed

**root reason:** reliability over speed. 300 domains over 1-3 hours is acceptable. 300 domains blocked is not.

**verdict:** correct choice. speed is not the constraint, success is.

---

## decision 3: cache layer for remote state

**why cache?**
1. why this approach? → because each scrape takes 3-5 seconds per domain
2. why? → because playwright navigation is inherently slow
3. why? → because we need to wait for network, render, and selectors
4. why? → because without cache, plan for 300 domains would take 15-25 minutes
5. why? → because fast plan iteration is critical for usability

**root reason:** user experience. fast feedback loops enable confident execution.

**verdict:** correct choice. without cache, the tool would be unusable for batch operations.

---

## decision 4: RefByUnique for domain references

**why RefByUnique?**
1. why this approach? → because domain objects need typed references
2. why? → because untyped bag refs cause bugs (absent properties, wrong keys)
3. why? → because intellisense should show what properties are needed
4. why? → because pit-of-success means correct usage should be easy
5. why? → because repo conventions require RefByUnique pattern

**root reason:** type safety and developer experience. the repo has established patterns.

**verdict:** correct choice. follows repo conventions and provides safety.

---

## decision 5: primary=['domain'] for child entities

**why parent reference in primary key?**
1. why this approach? → because dns records belong to domains, not standalone
2. why? → because RefByPrimary should answer "which parent owns this?"
3. why? → because unique key handles identity ("which specific record?")
4. why? → because this matches parent-child relationship semantics
5. why? → because domain-objects library expects meaningful primary keys

**root reason:** semantic correctness. primary key represents ownership, unique key represents identity.

**verdict:** correct choice. intentional design, not accident.

---

## decision 6: findsert/upsert mutation semantics

**why findsert and upsert?**
1. why this approach? → because idempotent operations are safe to retry
2. why? → because network failures and session timeouts happen
3. why? → because re-run of a batch should not cause duplicate actions
4. why? → because findsert returns found without mutation, upsert overwrites
5. why? → because these are the only safe mutation patterns for batch operations

**root reason:** reliability. batch operations must be resumable without side effects.

**verdict:** correct choice. idempotency is non-negotiable for batch automation.

---

## decision 7: separate DomainRegistration, DnsRecord, TransferRequest entities

**why three entities?**
1. why this approach? → because they have different lifecycles and operations
2. why? → because domains are queried, dns is read-only, transfers are requested
3. why? → because merge would create a god object
4. why? → because each has distinct unique keys and update semantics
5. why? → because separation enables targeted operations and cache

**root reason:** single responsibility. each entity represents one concept with one lifecycle.

**verdict:** correct choice. separation is cleaner than a monolithic domain entity.

---

## decision 8: totp secret required (no sms or passkey 2fa)

**why totp only?**
1. why this approach? → because totp can be automated with otpauth library
2. why? → because sms requires phone access, passkeys require hardware
3. why? → because automated batch operations need programmatic 2fa
4. why? → because manual 2fa would defeat the purpose of automation
5. why? → because users must configure their account for totp before use

**root reason:** automation requirement. only totp is automatable. constraint is documented.

**verdict:** correct choice. constraint is real and documented in vision.

---

## decision 9: typed status literals (not raw strings)

**why typed literals?**
1. why this approach? → because squarespace ui shows arbitrary text
2. why? → because raw strings would propagate through the system
3. why? → because operations need to switch on known status values
4. why? → because unknown statuses should be detected early
5. why? → because cast function normalizes text to canonical values

**root reason:** reliability. typed literals catch unexpected states at the boundary.

**verdict:** correct choice. added "reject" and "expir" patterns in assumptions review.

---

## decision 10: bot detection mitigation

**why these mitigations?**
1. why this approach? → because batch operations create detectable patterns
2. why? → because fixed intervals are obvious to bot detection
3. why? → because at 300 domains, patterns become highly visible
4. why? → because we need multiple mitigation layers
5. why? → because stealth plugin + random delays reduce detection risk

**mitigations implemented:**

1. **stealth plugin** — `stealthChromium.ts`:
   - masks `navigator.webdriver`
   - cleans HeadlessChrome from user agent
   - applies puppeteer-extra-plugin-stealth patches

2. **random delay** — `withNewLoggedInBrowserPage.ts` lines 83-86:
   - adds 200-500ms random delay after each operation
   - combined with bottleneck minTime of 500ms = 700-1000ms total

3. **captcha handle** (factory blueprint) — tiered approach:
   - checkbox click via headful pivot
   - human fallback for puzzle challenges

**verdict:** addressed with layered defenses. verified `stealthChromium.ts` exists (2026-03-12).

## why each decision holds

### decision 1 holds: playwright
**root:** squarespace has no api. not "doesn't have a good api" — has NO api whatsoever for domain operations. verified by:
- no api documentation in squarespace developer portal
- no api endpoints discoverable via network inspection
- community forums confirm no api exists

**why it holds:** there is literally no alternative. browser automation is the only path.

### decision 2 holds: sequential
**root:** rate limits and bot detection are real. verified by:
- squarespace uses cloudflare (see research brief)
- cloudflare detects parallel requests from single IP
- failed batch at domain 150 would be worse than slow batch

**why it holds:** reliability > speed for batch operations. 1-3 hours is acceptable. failure is not.

### decision 3 holds: cache
**root:** each scrape takes 3-5 seconds. verified by:
- `scrapeDomainsList.ts` does navigation + scroll + extract
- playwright network activity visible in traces
- 300 domains * 5 sec = 25 minutes per full scan

**why it holds:** without cache, `declastruct plan` would be unusable. fast feedback is critical.

### decision 4 holds: RefByUnique
**root:** repo conventions exist for good reason. verified by:
- `rule.forbid.dobj.bagrefs` brief documents why
- intellisense shows exactly what properties needed
- type errors catch wrong references at compile time

**why it holds:** type safety prevents bugs. pit-of-success makes correct usage easy.

### decision 5 holds: primary=['domain']
**root:** parent-child relationship semantics. verified by:
- dns records cannot exist without parent domain
- `RefByPrimary<DnsRecord>` answers "which domain owns this?"
- `RefByUnique<DnsRecord>` answers "which specific record?"

**why it holds:** semantic correctness. intentional design documented in blueprint.

### decision 6 holds: findsert/upsert
**root:** batch operations must be resumable. verified by:
- network failures happen (session timeout at domain 150)
- re-run should not create duplicate transfer requests
- findsert preserves, upsert overwrites — both idempotent

**why it holds:** idempotency is non-negotiable. re-run = safe.

### decision 7 holds: separate entities
**root:** different lifecycles and operations. verified by:
- domains are queried and mutated (unlock, dnssec)
- dns is read-only per wish scope
- transfers are requested (findsert only)

**why it holds:** single responsibility. god objects create complexity.

### decision 8 holds: totp only
**root:** automation requires programmatic 2fa. verified by:
- totp: `otpauth` library generates codes
- sms: requires phone, not automatable
- passkey: requires hardware, not automatable

**why it holds:** constraint is real. documented in vision as user requirement.

### decision 9 holds: typed literals
**root:** arbitrary ui text must be normalized. verified by:
- squarespace shows "In Progress", "Completed", "Code Sent"
- operations need to switch on known values
- unknown statuses should fail fast

**why it holds:** reliability. cast function normalizes at boundary.

### decision 10 holds: bot detection mitigation
**root:** batch operations create detectable patterns. verified by:
- `stealthChromium.ts` exists with StealthPlugin applied
- `withNewLoggedInBrowserPage.ts:83-86` has random 200-500ms delay
- factory blueprint completed captcha handle

**why it holds:** layered defenses reduce detection risk. verified 2026-03-12.
