### examined: selector fallback patterns

found fallback selectors in `src/access/sdks/squarespace.via.playwright/selectors/*.ts`:

```ts
// example from domainsListSelectors.ts
domainRow: '[data-test="domain-row"], .domain-row',
```

**analysis:**
these comma-separated selectors provide two options:
1. `[data-test="domain-row"]` — preferred stable selector
2. `.domain-row` — fallback class selector

**is this backwards compatibility?**

| characteristic | backcompat | robustness |
|----------------|-----------|------------|
| purpose | support old version of OUR code | anticipate EXTERNAL changes |
| direction | past-oriented | future-oriented |
| trigger | user upgrades our code | squarespace changes their ui |

**verdict:** these are **robustness patterns**, not backwards compatibility.
- they anticipate squarespace ui changes (future-oriented)
- they don't support old versions of declastruct (not past-oriented)
- explicitly designed in research: "implement selector fallback chains"

**no removal needed.** this is defensive code for external volatility, not internal backcompat.
