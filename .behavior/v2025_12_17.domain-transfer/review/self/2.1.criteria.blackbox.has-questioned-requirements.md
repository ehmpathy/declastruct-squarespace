# self-review: has-questioned-requirements

## the criteria

`.behavior/v2025_12_17.domain-transfer/2.1.criteria.blackbox.md`

## challenge: are any criteria unnecessary or misdirected?

for each usecase, i ask:
- who said this was needed? when? why?
- what if we removed this criterion — what would break?
- is this criterion scope too large or too small?
- could we test the same behavior more simply?

---

### usecase.1 = enumerate domains

**who said?** the wish: "get operations to see what domains are already in squarespace"

**if removed?** user cannot see current state. no plan possible. fundamental blocker.

**scope assessment**: appropriate. covers both populated and empty accounts.

**simpler test?** no — the given/when/then structure is already minimal.

**verdict**: necessary and well-scoped.

---

### usecase.2 = check dns records

**who said?** the wish: "get operations to see the dns of those domains"

**if removed?** user loses dns snapshot before transfer. service disruption risk.

**scope assessment**: appropriate. covers records present, absent, and invalid domain reference.

**simpler test?** no — three scenarios are the minimum for boundary coverage.

**verdict**: necessary and well-scoped.

---

### usecase.3 = unlock domain for transfer

**who said?** the wish implies transfer; vision explicitly states unlock is required.

**if removed?** domains remain locked. transfer impossible. fundamental blocker.

**scope assessment**: appropriate. covers normal unlock, idempotent re-unlock, and 60-day restriction.

**simpler test?** no — the three scenarios cover the critical paths.

**verdict**: necessary and well-scoped.

---

### usecase.4 = disable dnssec

**who said?** the vision: "disable dnssec" listed as step in before/after.

**if removed?** dnssec-enabled domains would fail transfer or cause dns resolution failure.

**scope assessment**: appropriate. covers enable→disable and already-disabled idempotent case.

**issue found**: missing a scenario.

**gap**: what if domain doesn't have dnssec support at all? should we test that disable on a non-dnssec domain is a no-op?

**resolution**: most domains support dnssec. if unsupported, the UI toggle wouldn't exist. the idempotent case ("already disabled") covers the "no action required" scenario. no change needed.

**verdict**: necessary and well-scoped.

---

### usecase.5 = request transfer authorization code

**who said?** the wish: "get/set operations to manage the export of domains"

**if removed?** user cannot get auth codes. transfer impossible. fundamental blocker.

**scope assessment**: appropriate. covers create, idempotent retry, and locked-domain error.

**simpler test?** no — three scenarios are the minimum for boundary coverage.

**verdict**: necessary and well-scoped.

---

### usecase.6 = batch domain transfer workflow

**who said?** the wish mentions "300+ domains to migrate" and "fully automated".

**if removed?** user would need to process domains one by one via explicit calls. the primary use case fails.

**scope assessment**: question — is "300 domains" the right number in the criterion?

**analysis**: the "300" is an example from the wish. the criterion tests batch behavior, not a specific count. the number is illustrative. any batch >1 exercises the same code paths.

**gap found**: the criterion says "domains are processed sequentially" but doesn't test the delay/rate-limit behavior.

**resolution**: rate-limit behavior is covered by usecase.8 (bot detection resilience). no duplication needed here.

**verdict**: necessary and well-scoped.

---

### usecase.7 = authentication

**who said?** the vision requires credentials for provider creation.

**if removed?** no way to verify auth works. users would discover auth failures only at runtime.

**scope assessment**: appropriate. covers totp success, sms rejection, invalid creds, and session refresh.

**issue found**: passkey 2fa not covered.

**gap**: the vision says squarespace supports totp, sms, and passkeys. the criteria cover totp and sms but not passkeys.

**resolution**: passkeys behave like sms — not automatable, should return same error. add passkey scenario.

**action taken**: updated criteria to add passkey scenario.

---

### usecase.8 = bot detection resilience

**who said?** the vision and premortem identify bot detection as a risk.

**if removed?** no verification that rate-limit and stealth measures work.

**scope assessment**: question — is this testable in a blackbox manner?

**analysis**: "operations complete without bot detection blocks" is outcome-based, not implementation-based. this is appropriate for blackbox. we cannot test "did stealth plugin run?" but we can test "did operations succeed?"

**gap found**: the criterion "execution pace avoids rate-based detection" is vague.

**resolution**: this is intentionally vague at blackbox level. the blueprint will specify the implementation (delays, randomization). blackbox only tests the outcome.

**verdict**: necessary and well-scoped.

## action taken

updated `2.1.criteria.blackbox.md` usecase.7 to add passkey scenario:

```
given('valid credentials with passkey 2fa')
  when('provider is created')
    then('returns error: totp required, passkeys not supported')
      sothat('user knows to reconfigure 2fa')
```

this aligns with the vision's constraint that only totp 2fa is supported.

## deeper consideration: what's missing from the criteria?

after the initial pass, i paused to consider: what scenarios are NOT covered?

### missing scenario 1: network failure mid-operation

**consideration**: what happens if network fails between unlock and transfer code request?

**analysis**: the criteria cover "batch fails mid-execution at domain 150" which includes network failures. the idempotent retry behavior handles this. no separate criterion needed.

### missing scenario 2: squarespace maintenance window

**consideration**: what if squarespace goes into maintenance mode mid-batch?

**analysis**: this would manifest as either:
- authentication failure (session invalid) — covered by usecase.7
- operation failure (page unavailable) — covered by batch retry in usecase.6
- bot detection trigger (cloudflare page) — covered by usecase.8

no separate criterion needed — extant criteria cover the symptoms.

### missing scenario 3: partial domain unlock

**consideration**: what if unlock succeeds but the UI doesn't reflect it immediately?

**analysis**: this is UI consistency, not a blackbox behavior. the criterion tests the outcome ("domain becomes unlocked"). if the outcome is verified via re-check, transient UI states don't matter.

**verdict**: no additional criteria needed. the 8 usecases provide sufficient coverage.
