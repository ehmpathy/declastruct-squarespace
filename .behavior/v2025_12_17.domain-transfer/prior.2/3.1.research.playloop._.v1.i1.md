# research: playloop feedback for declastruct-squarespace

## objective

identify what enables rapid playtest feedback for domain transfer wish. focus on experience reproduction and feedback speed.

---

## project type analysis

### classification: package + browser automation

this project is a **package** that:
- exposes declarative api surface (provider, daos)
- automates browser interactions (playwright)
- requires real remote state (squarespace account)

| project type | fundamental experience to reproduce |
|--------------|-------------------------------------|
| package | api surface, return values, error conditions |
| browser automation | dom state, navigation, scraped data |

the playtest loop must reproduce both:
1. developer experience: call api, get expected result
2. browser experience: page loads, selectors work, data scraped

---

## repo playtest capabilities (extant)

### test runner: jest

- configured via `jest.config.js`
- test:unit, test:integration, test:acceptance npm commands
- supports --testNamePattern for selective runs

### browser automation: playwright

- already integrated for scrape operations
- headless and headed modes supported
- wsEndpoint for reuse of browser instances

### bdd structure: test-fns

- given/when/then for test structure
- useBeforeAll for shared setup
- test output reads like specification

### credential guard: hasSquarespaceCredentials

- tests skip gracefully without credentials
- enables ci to run without secrets
- manual tests run with credentials

### sample data: getSampleSquarespaceContext

- creates real context from env vars
- enables integration tests against real account

### mock data: getMockedSquarespaceContext

- creates mocked context for unit tests
- no browser dependency

## playtest feedback loops

### loop 1: unit tests (instant feedback)

**trigger:** code change in domain logic
**time:** <5 seconds
**what it tests:** transforms, casts, validations

```
npm run test:unit -- castIntoDeclaredSquarespaceDomainRegistration.test.ts
```

**reproduction:** pure functions, no browser
**gap:** none - already fast

### loop 2: integration tests (slow feedback)

**trigger:** code change in scrapers
**time:** 30-60 seconds (browser launch + navigation)
**what it tests:** selectors work, scrape returns expected shape

```
npm run test:integration -- scrapeDomainsList.integration.test.ts
```

**reproduction:** real browser against real squarespace
**gap:** browser launch overhead

### loop 3: acceptance tests (slowest feedback)

**trigger:** full workflow validation
**time:** 2-5 minutes (multiple scrapes + mutations)
**what it tests:** full declastruct plan/apply workflow

```
npm run test:acceptance -- declastruct.acceptance.test.ts
```

**reproduction:** blackbox via declastruct cli
**gap:** requires full stack execution

## gaps and solutions

### gap 1: browser reuse between test runs

**problem:** each test run launches new browser (~5-10 seconds overhead)
**solution:** persistent browser via wsEndpoint

the repo already supports `browser.wsEndpointExtant` for browser reuse:

```ts
const provider = getDeclastructSquarespaceProvider({
  // ...credentials
  browser: { wsEndpointExtant: 'ws://localhost:9222/devtools/browser/xxx' },
});
```

**how to use for playtest:**
1. launch persistent browser: `npx playwright launch --headless --port 9222`
2. set wsEndpoint in test config
3. tests reuse browser session

### gap 2: visual debug for selector failures

**problem:** when selectors break, hard to see what changed
**solution:** playwright trace viewer

playwright captures traces that show:
- screenshots at each step
- dom state at each step
- network requests

```ts
await page.context().tracing.start({ screenshots: true, snapshots: true });
// ... test operations
await page.context().tracing.stop({ path: 'trace.zip' });
```

view with:
```sh
npx playwright show-trace trace.zip
```

[1] https://playwright.dev/docs/trace-viewer

### gap 3: cache invalidation debug

**problem:** cache makes tests pass but production fails (or vice versa)
**solution:** test with cache disabled

```ts
// in test setup
const cache = createOnDiskCache({
  directory: { mounted: { path: '.cache/test' } },
  expiration: { seconds: 0 }, // effectively disabled
});
```

or clear cache before test:
```sh
rm -rf .cache/squarespace
```

### gap 4: flaky selectors due to ui updates

**problem:** squarespace ui changes break selectors
**solution:** selector fallback pattern already in use

```ts
// already implemented
const domainsListSelectors = {
  domainRow: '[data-test="domain-row"], .domain-row', // primary, fallback
};
```

**playtest improvement:** add selector health check:
```ts
const checkSelectorHealth = async (page: Page) => {
  for (const [name, selector] of Object.entries(domainsListSelectors)) {
    const count = await page.$$(selector).length;
    console.log(`${name}: ${count} elements`);
  }
};
```

## playtest workflow

### developer iteration cycle

1. **change code** in scraper or domain operation
2. **run unit tests** for fast logic validation
3. **run single integration test** with persistent browser
4. **use trace viewer** if selector fails
5. **clear cache** if remote state is stale
6. **run full integration suite** before commit

### suggested npm command additions

```json
{
  "test:integration:watch": "jest --testPathPattern=integration --watch",
  "test:trace": "PLAYWRIGHT_TRACE=on npm run test:integration",
  "cache:clear": "rm -rf .cache/squarespace"
}
```

## reproduction fidelity

### what must be reproduced for playtest validity

| aspect | reproduction method |
|--------|---------------------|
| api surface | unit tests with mock context |
| return values | integration tests with assertions on shape |
| error conditions | unit tests + integration tests for edge cases |
| dom state | integration tests scrape real pages |
| navigation | integration tests follow real flows |
| scraped data | snapshot tests for data shape consistency |

### what does NOT need reproduction

| aspect | why skip |
|--------|----------|
| full 300 domain batch | time prohibitive; test with 1-5 domains |
| email delivery | out of scope; verify request was sent |
| transfer completion | 15-day process; not testable in playtest |

---

## conclusion

the repo already has comprehensive playtest infrastructure:
- jest + test-fns for bdd tests
- playwright for browser automation
- credential guard for ci/local separation
- sample/mock contexts for test isolation
- cache layer with configurable ttl

recommended improvements for faster playtest:
1. use persistent browser via wsEndpoint during development
2. add trace capture for selector debug
3. add cache:clear command for isolation
4. add test:watch for continuous feedback

no new tools needed. extant stack supports rapid playtest feedback.
