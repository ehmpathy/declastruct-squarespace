## icann domain transfer policy

### [1] icann registrant faq

**source:** https://www.icann.org/resources/pages/name-holder-faqs-2017-10-10-en

**key terms established:**
- authorization code (auth code, authinfo code, transfer code, epp code)
- registrar lock (transfer lock)
- 60-day transfer lock

**key quotes:**

> "You cannot transfer a domain name to a different registrar within 60 days of making changes to the registrant name, organization or email address"

> "You'll need the AuthInfo code (also called an Authorization Code, AuthInfo code, Auth-Info Code, or transfer code) to make the transfer. This is a unique code created by a registrar to identify the domain name holder and help prevent unauthorized transfers."

> "To transfer a domain, you need to unlock your domain by removing the registrar lock through your current provider's control panel"

**concepts leveraged:**
- domain transfer requires unlock + auth code
- 60-day lock is icann policy, not squarespace-specific
- auth code is unique per domain

### [2] auth-code wikipedia

**source:** https://en.wikipedia.org/wiki/Auth-Code

**key quotes:**

> "The registrar is required to provide the Auth-Code to the domain name owner within five calendar days of the owner's request."

**concepts leveraged:**
- registrars must provide auth code within 5 days

### [3] inter-registrar transfer policy

## dnssec technical references

### [7] rfc 4033 - dns security introduction

**source:** https://datatracker.ietf.org/doc/html/rfc4033

**concepts leveraged:**
- dnssec provides data origin authentication
- ds records at parent zone

### [8] google cloud dnssec migration

**source:** https://cloud.google.com/dns/docs/dnssec-migrate

**key quotes:**

> "To avoid resolution failures, disable DNSSEC before you transfer a domain name. DNSSEC relies on the DS record at the registrar to authenticate DNS responses. If you do not delete the DS record before the transfer, the chain of trust breaks. Recursive resolvers then reject the responses, which causes a service interruption."

**concepts leveraged:**
- dnssec must be disabled before transfer to prevent resolution failures
- ds record deletion required

### [9] cloudflare dnssec community

**source:** https://community.cloudflare.com/t/dnssec-not-turned-off-before-domain-transfer-cloudflare-showing-invalid-nameserve/776285

**key quotes:**

> "When you disable DNSSEC, it can take 24 to 48 hours for the process to completely finish. This is because of TTL (Time-to-Live) values, which control how long DNS records are cached."

**concepts leveraged:**
- 24-48 hour propagation after dnssec disable

## totp 2fa standards

### [10] rfc 6238 - totp algorithm

**source:** https://datatracker.ietf.org/doc/html/rfc6238

**key quotes:**

> "TOTP is defined as TOTP = HOTP(K, T), where T is an integer representing the number of time steps between the initial counter time T0 and the current Unix time, calculated as T = (Current Unix time - T0) / X."

> "TX (an interval which will be used to calculate the value of the counter, with a default of 30 seconds)"

> "RFC 6238 supports HMAC-SHA-1, HMAC-SHA-256, or HMAC-SHA-512."

**concepts leveraged:**
- totp uses 30-second intervals
- 6-digit codes standard
- sha1 is default algorithm

### [11] totp wikipedia

**source:** https://en.wikipedia.org/wiki/Time-based_one-time_password

**key quotes:**

> "Time-based one-time password (TOTP) is a computer algorithm that generates a one-time password using the current time as a source of uniqueness, and has been adopted as Internet Engineering Task Force (IETF) standard RFC 6238."

**concepts leveraged:**
- totp is ietf standard
- time-based password generation
