# research: root cause reflection — nameservers

## what is the symptom?

**observable problem**: cannot programmatically change domain nameservers on squarespace.

- **what is broken or absent?**: no API or SDK method to switch a domain from squarespace DNS to cloudflare (or back).
- **how does it manifest?**: users must manually navigate to each domain's nameserver settings page and click through the UI for every domain.
- **when does it occur?**: every time a user wants to migrate DNS provider for squarespace-registered domains.

---

## 5 whys tree

### branch 1: no programmatic access

1. why can't we programmatically change nameservers?
   → **squarespace has no domains API**

2. why does squarespace have no domains API?
   → **squarespace focuses on "closed system" for UX optimization; API is not a priority**

3. why hasn't someone built automation anyway?
   → **browser automation is complex; requires playwright/puppeteer expertise**

4. why isn't browser automation in this SDK yet?
   → **this SDK was built incrementally; nameservers feature not yet implemented**

5. why wasn't nameservers implemented earlier?
   → **[ROOT CAUSE]**: feature prioritization — other features (lock, dnssec, transfer) were implemented first; nameservers is next.

### branch 2: manual work is tedious

1. why is manual nameserver change tedious?
   → **must navigate to each domain's settings individually**

2. why can't you batch-change via squarespace UI?
   → **squarespace domains dashboard has no bulk actions for nameservers**

3. why does squarespace lack bulk nameserver actions?
   → **uncommon usecase for typical squarespace users (small business sites); not prioritized**

4. why do our users need bulk nameserver changes?
   → **our users are power users with domain portfolios, not typical squarespace users**

5. why are power users on squarespace?
   → **[ROOT CAUSE]**: squarespace acquired google domains; power users inherited domains without a choice.

---

## root cause summary

| layer | issue | is root cause? |
|-------|-------|----------------|
| surface | can't programmatically change nameservers | no — symptom |
| layer 1 | squarespace has no domains API | no — external constraint |
| layer 2 | browser automation not yet built | no — implementation gap |
| layer 3 | feature not yet prioritized | **yes** |
| alternate | power users inherited domains from google domains acquisition | **yes** (explains why need exists) |

---

## recommendation

**address root cause**: implement the nameserver feature now.

the root causes are:
1. **implementation gap** — feature exists conceptually in the SDK pattern but hasn't been built yet
2. **user need** — power users with domain portfolios need batch DNS provider migration

the fix is straightforward: follow extant patterns (domain object, DAO, get/set operations, scraper) to add nameserver support. this addresses both root causes:
- fills the implementation gap
- enables power users to migrate DNS providers programmatically

**no symptom treatment needed** — we are at the root cause by build of this feature.
