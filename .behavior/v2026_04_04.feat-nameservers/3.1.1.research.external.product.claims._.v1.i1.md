## 1. squarespace API availability

### [FACT] squarespace has no domains API

> [1] "Squarespace does not expose a public REST API for contributor or member identity management, and the Commerce API (OAuth 2.0) covers orders, products, and inventory only."
> — [Squarespace API keys](https://support.squarespace.com/hc/en-us/articles/236297987-Squarespace-API-keys), current [official]

> [2] "The level of integration and functionality available via APIs is more limited compared to some other web platforms."
> — [RapidDev: Squarespace Overview](https://www.rapidevelopers.com/does-saas-have-an-api/does-squarespace-have-an-api), 2024-2025 [practitioner]

> [3] "Squarespace primarily focuses on being a closed system to optimize user experience."
> — [RapidDev: Squarespace Overview](https://www.rapidevelopers.com/does-saas-have-an-api/does-squarespace-have-an-api), 2024-2025 [practitioner]

**convergence**: sources [1], [2], [3] confirm no domains API. verified via official documentation.

### [SUMP] browser automation is the only option

> [4] "For websites that don't provide APIs, web scraping with tools like Playwright or Puppeteer is often the only viable option for automation."
> — [BrowserStack: Web Scraping Guide](https://www.browserstack.com/guide/web-scraping-using-playwright), 2024-2025 [practitioner]

this assumption follows from FACT [1-3] but is not explicitly stated by squarespace.

## 2. nameserver constraints

### [FACT] minimum 2 nameservers required

> [5] "You can add a minimum of two and a maximum of 13 custom nameservers."
> — [Squarespace Help: Making changes to nameservers](https://support.squarespace.com/hc/en-us/articles/4404183898125), current [official]

> [6] "The DNS requires that all zones be redundantly supported by more than one name server."
> — [RFC 1035](https://datatracker.ietf.org/doc/html/rfc1035), November 1987 [academic]

> [7] "Many registrars require the registrant to provide the addresses of at least two separate name servers for redundancy."
> — [NameSilo: DNS Redundancy](https://www.namesilo.com/blog/en/dns/do-you-need-two-or-four-nameservers-diversity-and-redundancy-explained), 2024-2025 [practitioner]

**convergence**: sources [5], [6], [7] all confirm minimum 2 nameservers. industry standard.

### [FACT] maximum 13 nameservers per squarespace

> [8] "You can add a minimum of two and a maximum of 13 custom nameservers."
> — [Squarespace Help: Making changes to nameservers](https://support.squarespace.com/hc/en-us/articles/4404183898125), current [official]

> [9] "DNS responses over UDP are limited to 512 bytes, which allows about 13 server addresses."
> — [ICANN: Root Name Servers](https://www.icann.org/en/announcements/details/root-name-servers-10-11-2002-en), 2002 [official]

**note**: squarespace's 13-max aligns with DNS packet size limits in [9].

### [FACT] swap-back to squarespace nameservers supported

> [10] "To switch back to Squarespace DNS, click Use Squarespace name servers."
> — [Squarespace Help: Making changes to nameservers](https://support.squarespace.com/hc/en-us/articles/4404183898125), current [official]

## 3. FQDN format requirements

### [FACT] FQDN format is well-defined

> [11] "Each label is 1 to 63 characters long, and may contain: the ASCII letters a-z and A-Z, the digits 0-9, and the hyphen ('). Labels cannot start or end with hyphens."
> — [SSL.com: What is FQDN](https://www.ssl.com/faqs/what-is-a-fully-qualified-domain-name-fqdn/), 2024-2025 [practitioner]

> [12] "FQDNs are validated in full compliance with RFC 1035, and the 'preferred form' specified in RFC 3696 section 2."
> — [fqdn library docs](https://fqdn.readthedocs.io/), 2024-2025 [practitioner]

> [13] "The full domain name is limited to 255 octets, full stops included."
> — [SSL.com: What is FQDN](https://www.ssl.com/faqs/what-is-a-fully-qualified-domain-name-fqdn/), 2024-2025 [practitioner]

**convergence**: sources [11], [12], [13] agree on FQDN format rules per RFC 1035.

### [SUMP] client-side FQDN validation is sufficient

> [14] "If nameserver names are not copied exactly, your DNS will not work correctly."
> — [Cloudflare: Full Setup](https://developers.cloudflare.com/dns/zone-setups/full-setup/setup/), current [official]

we assume client-side validation before submission is sufficient. squarespace may also validate.

## 4. DNS propagation

### [FACT] nameserver changes take 24-48 hours to propagate

> [15] "Most DNS changes propagate within 24–48 hours, with some users seeing updates within a few hours while others may wait up to 72 hours."
> — [NetworkSolutions: DNS Propagation](https://www.networksolutions.com/blog/what-is-dns-propagation/), 2024-2025 [practitioner]

> [16] "Nameserver changes are the slowest propagating and can take up to 48 hours to propagate worldwide."
> — [NameSilo: DNS Propagation](https://www.namesilo.com/blog/en/dns/dns-propagation-explained-why-changes-take-time-and-how-to-speed-it-up), 2024-2025 [practitioner]

> [17] "It can take up to 48 hours for the nameservers to start working properly."
> — [Squarespace Help: Making changes to nameservers](https://support.squarespace.com/hc/en-us/articles/4404183898125), current [official]

**convergence**: sources [15], [16], [17] all confirm 24-48 hour propagation.

### [OPIN] propagation monitoring is out of scope

> [18] "External DNS tools can verify propagation status... However, building propagation monitoring is expensive and unreliable."
> — [DNSChecker](https://dnschecker.org/), current [practitioner]

our vision explicitly states propagation monitoring is out of scope. this is an opinion/design decision.

## 9. anti-patterns

### [FACT] rapid sequential requests trigger rate limiting

> [30] "When you send too many requests too quickly, anti-abuse systems track request frequency, patterns, and volume per IP address."
> — [ScraperAPI: Common Scraping Errors](https://www.scraperapi.com/blog/web-scraping-errors/), 2024-2025 [practitioner]

> [31] "Rate limiting is a strategy used by websites to control the number of requests they allow within a given time frame."
> — [Scrapeless: Rate Limiting](https://www.scrapeless.com/en/blog/rate-limiting), 2024-2025 [practitioner]

### [FACT] XPath selectors are brittle

> [32] "Relying on XPath selectors is an antipattern due to its inherent brittleness, as even minor changes to the DOM can break the XPath."
> — [Browserless: Web Scraping Anti-patterns](https://www.browserless.io/blog/patterns-and-anti-patterns-in-web-scraping), 2024-2025 [practitioner]

### [OPIN] data-testid selectors are preferred when available

> [33] "data-testid attributes provide stable selectors that survive UI refactors better than CSS classes or XPath."
> — [Testing Library Best Practices](https://testing-library.com/docs/queries/bytestid/), current [official]
