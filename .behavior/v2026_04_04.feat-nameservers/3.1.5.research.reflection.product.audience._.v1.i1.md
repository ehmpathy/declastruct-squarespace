# research: audience reflection — nameservers

## who is the audience?

### primary: domain portfolio managers

operators who manage 10-300+ domains registered with squarespace and need to migrate DNS management to cloudflare (or another external provider) for:
- DDoS protection
- edge cache
- advanced DNS features
- consolidated DNS management across providers

typically: devops engineers, sysadmins, agency operators, or startup founders with multiple brands.

### secondary: developers

developers who integrate `declastruct-squarespace` into their infrastructure-as-code pipelines. they want:
- programmatic control over DNS provider selection
- idempotent operations for CI/CD workflows
- clear error messages for automation

### tertiary: future users who evaluate the SDK

users who browse the SDK to see if it supports their usecase. they want to know:
- can I point my squarespace domains to cloudflare?
- can I automate this across many domains?
- is the operation reversible?

## what do they care about?

### criteria for success

| audience | criteria |
|----------|----------|
| portfolio managers | operation completes without error, is idempotent, is reversible |
| developers | clear API, consistent with other operations, good error messages |
| evaluators | documentation, examples, test coverage |

### tradeoffs they accept

- **accept**: 24-48 hour DNS propagation delay (out of our control)
- **accept**: one domain at a time (not batch API, but can loop)
- **reject**: silent failures or partial state
- **reject**: inability to swap back to squarespace default

### delight vs disappointment

| delight | disappointment |
|---------|----------------|
| operation is fast (seconds for no-op, minutes for change) | slow operations (30+ seconds) with no feedback |
| clear feedback on what changed | ambiguous success/failure |
| works on first try | requires retry or manual intervention |
| reversible (swap back to squarespace) | one-way operation |

## how will they experience it?

### touchpoints

1. **discovery**: user reads SDK docs, sees `DeclaredSquarespaceDomainNameservers` entity
2. **comprehension**: user reads vision doc or README examples
3. **implementation**: user calls `setNameservers({ upsert: { domain: { name }, nameservers: [...] } })`
4. **verification**: user calls `getNameservers` or uses external DNS lookup tool
5. **reversal** (optional): user calls `setNameservers({ upsert: { domain: { name }, nameservers: null } })`

### friction points

| friction | mitigation |
|----------|------------|
| minimum 2 nameservers required | clear validation error with RFC citation |
| FQDN format validation | clear error with example format |
| DNS propagation delay | documentation that propagation is external (24-48h) |
| squarespace UI changes | centralized selectors, integration tests catch drift |

### smooth experience

1. user imports `DeclaredSquarespaceDomainNameserversDao`
2. user calls `Dao.set.upsert({ domain: { name: 'example.com' }, nameservers: ['ns1.cloudflare.com', 'ns2.cloudflare.com'] })`
3. operation completes in <60 seconds
4. user receives entity with updated nameservers
5. user verifies via DNS lookup after propagation
