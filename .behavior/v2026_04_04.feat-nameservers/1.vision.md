# vision: domain nameservers support

## the outcome world

### before
- domains on squarespace use squarespace's nameservers by default
- to use cloudflare (or another DNS provider), you must manually navigate to each domain's nameserver settings and enter custom nameservers
- no programmatic way to batch-change nameservers across many domains
- no way to track whether a domain is using squarespace DNS or external DNS

### after
- declarative control over domain nameservers via `DeclaredSquarespaceDomainNameservers` entity
- `DeclaredSquarespaceDomainNameserversDao` for declarative management via `genDeclastructDao`
- `setNameservers({ upsert: { domain: { name: 'example.com' }, nameservers: ['ns1.cloudflare.com', ...] } })` swaps to cloudflare
- `setNameservers({ upsert: { domain: { name: 'example.com' }, nameservers: null } })` swaps back to squarespace default
- batch operations across 100+ domains become feasible (loop over setNameservers)
- nameserver state tracked as separate entity, consistent with `DeclaredSquarespaceDomainDnsRecord`

### the "aha" moment
when you need to migrate 50 domains from squarespace DNS to cloudflare for better performance/protection, you run one operation per domain instead of clicking through 50 separate domain settings pages.

---

## user experience

### usecases

1. **migrate to cloudflare**: user wants all their domains to use cloudflare DNS for DDoS protection and edge caching
2. **migrate back to squarespace**: user realizes they don't need cloudflare and wants to simplify by returning to squarespace DNS
3. **audit DNS providers**: user wants to know which domains use which DNS provider across their portfolio

### contract inputs & outputs

```typescript
// domain object
interface DeclaredSquarespaceDomainNameservers {
  /**
   * .what - Reference to the domain this nameserver config belongs to
   */
  domain: RefByUnique<typeof DeclaredSquarespaceDomainRegistration>;

  /**
   * .what - Custom nameservers for this domain
   * .why - Allows use of external DNS providers like cloudflare
   * .note - null = squarespace default nameservers, [...] = custom nameservers
   */
  nameservers: string[] | null;
}

// DAO for declarative management
const DeclaredSquarespaceDomainNameserversDao = genDeclastructDao<
  typeof DeclaredSquarespaceDomainNameservers,
  ContextSquarespaceAgent
>({
  dobj: DeclaredSquarespaceDomainNameservers,
  get: {
    one: {
      byUnique: async (input, context) => getNameservers({ by: { unique: input } }, context),
      byPrimary: null,
    },
  },
  set: {
    findsert: async (input, context) => setNameservers({ findsert: input }, context),
    upsert: async (input, context) => setNameservers({ upsert: input }, context),
    delete: null, // nameserver config cannot be deleted, only reset to null
  },
});

// get current nameservers for a domain
const ns = await getNameservers({ by: { unique: { domain: { name: 'example.com' } } } }, context);
// => { domain: { name: 'example.com' }, nameservers: null } // squarespace default
// => { domain: { name: 'example.com' }, nameservers: ['ns1.cloudflare.com', 'ns2.cloudflare.com'] } // custom

// swap to cloudflare
await setNameservers({
  upsert: {
    domain: { name: 'example.com' },
    nameservers: ['ns1.cloudflare.com', 'ns2.cloudflare.com'],
  }
}, context);

// swap back to squarespace
await setNameservers({
  upsert: {
    domain: { name: 'example.com' },
    nameservers: null, // squarespace manages automatically
  }
}, context);

// or via DAO for declarative management
await DeclaredSquarespaceDomainNameserversDao.set.upsert(
  { domain: { name: 'example.com' }, nameservers: ['ns1.cloudflare.com', 'ns2.cloudflare.com'] },
  context,
);
```

### timeline

1. user calls `getNameservers` to see current nameserver config
2. user calls `setNameservers` with desired nameservers
3. squarespace UI is automated to make the change
4. user verifies with `getNameservers` (or DNS lookup) that change propagated

## mental model

### how users would describe this
> "i can programmatically point my squarespace domains to cloudflare or back to squarespace DNS"

### analogies
- like a DNS registrar's "change nameservers" panel, but automated
- like terraform for DNS provider selection

### terms

| user term | our term |
|-----------|----------|
| "point to cloudflare" | `setNameservers({ upsert: { domain: { name }, nameservers: ['ns1.cloudflare.com', ...] } })` |
| "use squarespace DNS" | `setNameservers({ upsert: { domain: { name }, nameservers: null } })` |
| "custom nameservers" | `nameservers: [...]` (non-null array) |
| "default nameservers" | `nameservers: null` |

## evaluation

### pros
- declarative, idempotent control over nameservers
- follows extant patterns (separate entity, DAO via `genDeclastructDao`, get/set operations)
- clean separation of concerns — nameservers are not domain registration metadata
- full test coverage ensures reliability

### cons
- DNS propagation takes time (24-48h); our API can't speed that up
- squarespace UI may change; scraper maintenance required

### edgecases & pit of success

| edgecase | handling |
|----------|----------|
| domain locked | fail fast with clear error before attempting change |
| invalid nameserver format | validate NS records format before submission |
| partial nameserver list | require at least 2 nameservers for CUSTOM |
| nameserver change pending | detect and report propagation status if available |

## open questions & assumptions

### assumptions
- squarespace's nameserver settings page at `/dns/domain-nameservers` follows similar patterns to other domain detail pages
- changing nameservers doesn't require additional reauthentication beyond the standard session
- squarespace allows switching back to their default nameservers after using custom

### answered questions
1. **propagation status**: out of scope. "change submitted" is sufficient. DNS propagation monitor is expensive and external DNS tools can verify if needed.
2. **nameserver validation**: validate FQDN format only. do not validate that nameservers actually work (reachable, configured) — too expensive and unreliable.

### external research needed
- scrape the `/dns/domain-nameservers` page to understand UI structure and selectors
- verify what confirmation dialogs appear when changing nameservers
- understand what error states exist (e.g., invalid nameserver, locked domain)

## what is awkward?

### cloudflare-specific vs generic
the wish mentions "cloudflare" specifically, but the feature is really "custom nameservers". the design is generic — any nameservers can be specified, with cloudflare as an example in docs.

### null vs empty array semantics
- `null` = squarespace default nameservers
- `[]` (empty array) = ambiguous — could mean "no nameservers" which is invalid

**decision**: treat empty array same as null (squarespace default). validate that custom nameservers have at least 2 entries.
