# research: premortem reflection — nameservers

## premortem: imagine we shipped and it failed

project forward: we shipped this, and it failed badly. why?

### most plausible failure scenarios

1. **squarespace UI changed**: selectors broke, scraper fails silently or throws cryptic errors. users report "it worked last week" but now fails.

2. **nameserver validation too strict**: valid nameservers rejected because FQDN regex doesn't cover edge cases (punycode, new TLDs, unusual patterns).

3. **reauthentication flow unhandled**: squarespace prompts for password/2FA when you change nameservers, but our scraper doesn't detect it. operation hangs or fails without clear reason.

4. **cloudflare nameserver format differs from expected**: cloudflare changed their nameserver name convention (e.g., from `ns1.cloudflare.com` to `dns1.cloudflare.com`), and users can't figure out why validation fails.

5. **swap-back doesn't actually work**: we implemented set-to-null as "reset to default" but squarespace's "Use Squarespace name servers" button does a different action. users get stuck with custom nameservers.

---

## what obvious signs did we ignore?

- **no actual UI exploration**: we assumed the `/dns/domain-nameservers` page follows similar patterns to other pages, but we haven't scraped it yet to verify.

- **confirmation dialogs unknown**: we don't know what confirmation dialogs appear when you change nameservers. could be multi-step, could require enter domain name to confirm.

- **rate limits not tested**: squarespace may rate-limit nameserver changes more aggressively than other operations since they affect DNS resolution.

- **propagation confusion**: users might think setNameservers failed because DNS lookup still shows old nameservers (24-48h propagation). no clear "operation submitted successfully, await propagation" feedback.

---

## what assumptions could turn out wrong?

| assumption | what if wrong? | likelihood |
|------------|----------------|------------|
| nameserver page at `/dns/domain-nameservers` follows similar UI patterns | selectors don't work, need custom scraper logic | MEDIUM |
| no reauthentication required for nameserver changes | scraper hangs or fails mysteriously | MEDIUM |
| squarespace validates nameserver format client-side | server rejects valid nameservers with unhelpful error | LOW |
| "Use Squarespace name servers" button resets to default | button does a different action (e.g., opens support article) | LOW |
| minimum 2 nameservers is enforced by squarespace | squarespace allows 1 nameserver, our validation blocks valid config | LOW |
| maximum 13 nameservers per RFC/squarespace docs | actual limit differs | LOW |

---

## what is the nightmare scenario?

**scenario**: user migrates 100 domains to cloudflare nameservers with our SDK. squarespace UI changed silently, and instead of custom nameservers, the scraper accidentally clicks "Transfer domain" or "Delete DNS records" due to similar button selectors. user loses DNS configuration for 100 domains, sites go offline for 24-48 hours as propagation settles.

**what makes this a nightmare**:
- **scale**: batch operation affects many domains
- **irreversibility**: DNS changes propagate globally, can't be instantly rolled back
- **visibility**: sites go offline, customers/users notice immediately
- **trust**: user loses confidence in the SDK, won't use it again

---

## what mitigations to consider?

| risk | mitigation | cost of mitigation |
|------|------------|--------------------|
| UI changed, selectors break | integration tests run against live squarespace account; CI catches drift | LOW — already have this pattern |
| reauthentication unhandled | reuse `handleReauthentication` pattern from toggleDomainLock | LOW — pattern exists |
| wrong button clicked due to selector drift | use data-testid selectors first, text selectors as fallback; verify URL after navigation | LOW |
| user confused by propagation delay | documentation and return value indicate "submitted, await 24-48h propagation" | LOW |
| validation too strict | use lenient FQDN validation (allow punycode, new TLDs); test against cloudflare, route53, namecheap nameservers | LOW |
| swap-back doesn't work | integration test specifically for null → squarespace default flow | LOW |
| nightmare scenario (mass DNS destruction) | verify page URL before any mutation; fail-fast if URL doesn't match `/dns/domain-nameservers` | LOW |
