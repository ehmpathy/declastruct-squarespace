# research: external product access — nameservers

## executive summary

squarespace does not expose a domains API. nameserver management must be automated via browser (playwright). the UI at `/dns/domain-nameservers` allows setting custom nameservers or resetting to squarespace defaults. minimum 2, maximum 13 nameservers per squarespace's constraints. DNS propagation takes 24-48 hours.

---

> [1] "Squarespace primarily focuses on being a closed system to optimize user experience, meaning its API capabilities might not be as broad or flexible as open-source platforms like WordPress."

> [2] "Squarespace does not expose a public REST API for contributor or member identity management, and the Commerce API (OAuth 2.0) covers orders, products, and inventory only."
> — [Squarespace API keys](https://support.squarespace.com/hc/en-us/articles/236297987-Squarespace-API-keys), current [official]

> [3] "The level of integration and functionality available via APIs is more limited compared to some other web platforms."
> — [RapidDev: Squarespace Overview](https://www.rapidevelopers.com/does-saas-have-an-api/does-squarespace-have-an-api), 2024-2025 [practitioner]

**convergence**: sources [1], [2], [3] all confirm no domain management API exists. browser automation is required.

## 3. DNS nameserver requirements (RFC standards)

### finding: minimum 2 nameservers required

> [8] "RFC 1035 is the revised specification of the protocol and format used in the implementation of the Domain Name System. The DNS requires that all zones be redundantly supported by more than one name server."
> — [RFC 1035](https://datatracker.ietf.org/doc/html/rfc1035), November 1987 [academic]

> [9] "The DNS has three major components: The DOMAIN NAME SPACE and RESOURCE RECORDS, which are specifications for a tree structured name space and data associated with the names."
> — [RFC 1034](https://datatracker.ietf.org/doc/html/rfc1034), November 1987 [academic]

> [10] "Many registrars require the registrant to provide the addresses of at least two separate name servers for redundancy—when a domain name is served by two name servers, name resolution will continue even if one of them goes down."
> — [NameSilo: DNS Diversity and Redundancy](https://www.namesilo.com/blog/en/dns/do-you-need-two-or-four-nameservers-diversity-and-redundancy-explained), 2024-2025 [practitioner]

> [11] "It is recommended that zone operators follow guidelines which recommend that multiple name servers be configured for each zone and that the name servers be separated both physically and via connectivity routes."
> — [RFC 6168](https://www.rfc-editor.org/rfc/rfc6168.html), May 2011 [academic]

**convergence**: RFC standards [8], [9], [11] and practitioner source [10] all require minimum 2 nameservers. aligns with squarespace's minimum of 2 per source [5].

## 4. FQDN validation patterns

### finding: FQDN format rules

> [12] "Each label is 1 to 63 characters long, and may contain: the ASCII letters a-z and A-Z, the digits 0-9, and the hyphen ('). Labels cannot start or end with hyphens (RFC 952) and labels can start with numbers (RFC 1123). The full domain name is limited to 255 octets, full stops included."
> — [SSL.com: What is FQDN](https://www.ssl.com/faqs/what-is-a-fully-qualified-domain-name-fqdn/), 2024-2025 [practitioner]

> [13] "FQDNs are validated in full compliance with RFC 1035, and the 'preferred form' specified in RFC 3696 section 2."
> — [fqdn library docs](https://fqdn.readthedocs.io/), 2024-2025 [practitioner]

> [14] "An FQDN specifies its exact location in the tree hierarchy of the Domain Name System (DNS). It specifies all domain levels, including the top-level domain and the root zone."
> — [F5: What is FQDN](https://www.f5.com/glossary/fully-qualified-domain-name-fqdn), 2024-2025 [practitioner]

**convergence**: sources [12], [13], [14] agree on FQDN format: labels 1-63 chars, alphanumeric + hyphen, total max 255 chars.

## 5. DNS propagation timing

### finding: 24-48 hours typical propagation

> [15] "Most DNS changes propagate within 24–48 hours, with some users seeing updates within a few hours while others may wait up to 72 hours, depending on their ISP's caching policies."
> — [NetworkSolutions: DNS Propagation](https://www.networksolutions.com/blog/what-is-dns-propagation/), 2024-2025 [practitioner]

> [16] "Nameserver changes are the slowest propagating and can take up to 48 hours to propagate worldwide, because nameservers' records are retrieved from root nameservers, which give out large TTL values of 2 days or more."
> — [NameSilo: DNS Propagation](https://www.namesilo.com/blog/en/dns/dns-propagation-explained-why-changes-take-time-and-how-to-speed-it-up), 2024-2025 [practitioner]

> [17] "At least 24–48 hours before making your DNS change, reduce your TTL to 300 seconds (5 minutes), which instructs DNS resolvers worldwide to refresh your records every five minutes, rather than every few hours."
> — [Varonis: DNS TTL Best Practices](https://www.varonis.com/blog/dns-ttl), 2024-2025 [practitioner]

> [18] "TTL (time to live) represents the time each step takes for DNS to cache a record. The TTL tells DNS resolvers how long they should cache a DNS record before requesting a fresh copy from the authoritative DNS server."
> — [Cloudflare TTL docs](https://developers.cloudflare.com/dns/manage-dns-records/reference/ttl/), current [official]

**convergence**: sources [15], [16], [17], [18] all confirm 24-48 hour propagation for nameserver changes. this is outside our control.

**convergence**: sources [19], [20], [21], [22] all recommend storageState for session reuse. **anti-pattern**: source [22] warns against committing auth state to git.

### finding: headless detection signals

> [24] "playwright-stealth is a Python package that extends Playwright by overriding specific configurations to avoid bot detection. It deletes the navigator.webdriver property and removes 'HeadlessChrome' from the User-Agent header set by Chrome in headless mode by default."
> — [ScrapeOps: Make Playwright Undetectable](https://scrapeops.io/playwright-web-scraping-playbook/nodejs-playwright-make-playwright-undetectable/), 2024-2025 [practitioner]

> [25] "Bot detection systems don't detect Playwright directly—they detect non-human patterns, which usually come from how automation behaves, not from the tool itself."
> — [BrowserStack: Playwright Bot Detection](https://www.browserstack.com/guide/playwright-bot-detection), 2024-2025 [practitioner]

> [26] "You can mimic human behavior when interacting with a website by introducing realistic delays, random mouse movements and natural scrolling. Real users hesitate, scroll, pause, and interact inconsistently, while automated flows that click and type instantly with perfect precision are easy to identify."
> — [DEV: Headless Browser Evasion 2026](https://dev.to/vhub_systems_ed5641f65d59/how-sites-detect-headless-browsers-and-how-to-evade-each-signal-2026-guide-2jj0), 2026 [practitioner]

## 9. anti-patterns and warnings

### finding: rate limiting

> [30] "Modern websites protect themselves from aggressive scrapers, and when you send too many requests too quickly, anti-abuse systems track request frequency, patterns, and volume per IP address, creating scraping errors that can stop data collection."
> — [ScraperAPI: Common Scraping Errors](https://www.scraperapi.com/blog/web-scraping-errors/), 2024-2025 [practitioner]

> [31] "Rate limiting is a strategy used by websites to control the number of requests they allow within a given time frame. If your scraper exceeds this limit, the server responds with a '429 Too Many Requests' status code."
> — [Scrapeless: Rate Limiting](https://www.scrapeless.com/en/blog/rate-limiting), 2024-2025 [practitioner]

> [32] "IP Rate Limiting is the #1 way of rate limiting, simply associating the number of requests with the user's IP address. Web Application Firewall services like Cloudflare, Akamai, DataDome, and PerimeterX implement sophisticated rate-limiting mechanisms."
> — [Apify: Rate Limiting Academy](https://docs.apify.com/academy/anti-scraping/techniques/rate-limiting), 2024-2025 [practitioner]

### finding: selector brittleness

> [33] "Relying on XPath selectors is an antipattern due to its inherent brittleness, as even minor changes to the DOM can break the XPath. Failing to block unnecessary resources such as images, CSS files, and fonts is a common antipattern that significantly slows down scraping."
> — [Browserless: Web Scraping Anti-patterns](https://www.browserless.io/blog/patterns-and-anti-patterns-in-web-scraping), 2024-2025 [practitioner]

**warnings from sources [30], [31], [32], [33]**:
- avoid rapid sequential requests (rate limiting)
- avoid XPath selectors (brittle)
- use data-testid or stable selectors when available
- block unnecessary resources for performance
